#### LOW-10.9-2 — Regex `^def check_project_model_enabled\b` manque re-définitions **indentées** (nested/class method)

**Fichier** : `backend/tests/test_core/test_feature_flags.py:207`

**Symptôme** : la regex `^def check_project_model_enabled\b` (ancre `^`) ne match que les définitions **module-level** (colonne 0). Un dev pourrait introduire par erreur :

```python
class SomeRouter:
    def check_project_model_enabled(self):  # ← indenté, test ne le détecte pas
        ...
```

…ou une redéfinition nested dans un décorateur, ce qui contournerait l'assertion `len(hits) == 1`.

**Impact** : faible. Une re-définition en méthode de classe portant le même nom serait un anti-pattern évident au review (et n'aurait pas la signature dépendance FastAPI attendue). Mais le test prétend enforcer « 1 seule définition » sans couvrir ce cas.

**Recommandation** : remplacer `r"^def check_project_model_enabled\b"` par `r"^\s*def check_project_model_enabled\b"` (tolère toute indentation). Correction triviale. Alternative AST-based plus rigoureuse (`ast.parse` + walk `FunctionDef.name == "check_project_model_enabled"`) pour les cas extrêmes, mais over-engineering pour Story S.

#### LOW-10.9-3 — Absence de test automatisé pour coercion Pydantic `ENABLE_PROJECT_MODEL="garbage"` → `ValidationError`

**Fichier** : `backend/tests/test_core/test_feature_flags.py` (manquant)

**Symptôme** : la Task 4.2 vérifie **manuellement** que `ENABLE_PROJECT_MODEL=garbage python -c "from app.core.config import Settings; Settings()"` lève `ValidationError`. **Aucun test automatisé** ne verrouille cette assertion (AC2 clause 2).

**Impact** : si Pydantic v2 change son comportement de coercion `bool` dans une montée mineure (v2.11 → v2.12), la coercion pourrait accepter silencieusement `"garbage"` en `False` ou `True` sans détection CI. Faible probabilité (stable Pydantic v2), mais l'AC2 mentionne explicitement « Pydantic coerce `ENABLE_PROJECT_MODEL="garbage"` → erreur de boot explicite » et ce n'est pas testé.

**Recommandation** : ajouter un test `test_settings_rejects_garbage_bool_value(monkeypatch)` :

```python
@pytest.mark.unit
def test_settings_rejects_garbage_bool_value(monkeypatch):
    from pydantic import ValidationError
    monkeypatch.setenv("ENABLE_PROJECT_MODEL", "garbage-value")
    from app.core.config import Settings
    with pytest.raises(ValidationError):
        Settings()
```

Ajout 6 lignes, durcit AC2. Peut être inclus dans le prochain patch batch.

## Focus Spécifique 10.9 — Analyse des 5 points remontés par le reviewer requestor

### 1. Pivot `Path.rglob` vs `subprocess rg` — scan aussi complet ? performance acceptable ?

**Complétude** : ✅ **Équivalent fonctionnel**.
- `Path.rglob("*.py")` parcourt récursivement `backend/app/` et match tous les `.py`.
- `rg -n "pattern" backend/app/` aurait matché les mêmes fichiers (rg exclut `.gitignore` par défaut, mais `backend/app/` n'a pas de `.gitignore` interne ignorant du code Python).
- Couverture **identique** pour le périmètre review.

**Performance** : ✅ **Acceptable**.
- `backend/app/` ~150-250 fichiers Python (~2-3 MB lecture).
- `Path.rglob + read_text` : ~50-200 ms single-threaded. Négligeable dans la suite pytest (203 s au total).
- `rg` aurait été ~10× plus rapide (multi-thread + SIMD) mais le différentiel absolu est non-mesurable à l'échelle test.

**Portabilité** : ✅ **Supérieure à subprocess rg** (pas de dépendance binaire, fonctionne en CI headless stock Python 3.12/3.14). Justifie le pivot Dev Agent Record.

### 2. Task 8 commits intermédiaires facultatifs non exécutée — justifié pour story atomique S ?

**Oui** ✅. Critères leçon 10.8 :
- Diff total < 300 lignes ? **Vérifié** (config.py +16, feature_flags.py +19/-3, router.py +1/-10, test_feature_flags.py +175, docs 2 nouveaux fichiers) = ~200 lignes ajoutées.
- Refactor "pur git mv" isolable ? Oui (Task 2). Mais le diff est petit et lisible d'un coup.
- Commit atomique facilite le rollback ? Oui à cette taille.

La Task 8.1 (split commit 1 refactor + commit 2 formalisation) aurait **ajouté du bruit** sans valeur review (1 commit intermédiaire de 25 lignes vs 1 commit final atomique de 200 lignes). Le skip est **conforme à la spec** (Tasks /Subtasks §Task 8 marquée `facultatif si la PR finale reste atomique et < 500 lignes de diff`).

### 3. `enable_project_model` champ informationnel — risque de consommation par erreur ?

**Risque mitigé** par **3 couches de défense** :
1. **Test enforcement** `test_no_applicative_caller_reads_settings_enable_project_model` scan 0 hit dans `backend/app/` (ligne 82-105). Une PR qui introduit `settings.enable_project_model` dans du code applicatif casse CI immédiatement.
2. **Commentaire bloc** config.py:105-111 explique explicitement le rôle informationnel.
3. **Field description** mentionne « Lu dynamiquement par is_project_model_enabled() au runtime ; le champ Settings est informationnel. »

**Résiduel** : friction DX (cf. LOW-10.9-4) — le nom de champ ne porte aucun signal « informationnel ». Un dev verra l'échec CI plutôt qu'une signature auto-explicative. **Non-bloquant**.

**Verdict** : architecture correcte, défense en profondeur suffisante. La suggestion LOW-10.9-4 est amélioration DX facultative.

### 4. Marker CLEANUP Story 20.1 — test enforcement détecte bien le marker ?

**Oui** ✅. Le test `test_feature_flags_has_cleanup_marker` (test_feature_flags.py:148-170) vérifie **2 invariants textuels indépendants** :
- `"CLEANUP: Story 20.1" in content` — identifiant de cleanup
- `"migration 027" in content` — référence migration

Les 2 chaînes sont présentes en `feature_flags.py:13-14`. Robuste contre suppression accidentelle du marker (PR qui nettoie « commentaires inutiles » → CI casse). Robuste contre renommage partiel (si un dev remplace `Story 20.1` par `Story 21.0` en oubliant d'updater la migration, l'assertion `migration 027` tient encore tant que la migration ne change pas).

**Limite connue** : si Story 20.1 devient Story 20.2 ET migration 027 devient migration 028 en même temps (rename coordonné), le test passerait avec les deux nouvelles valeurs si le dev update le test. Mais c'est **by design** — un rename coordonné est une décision Tech Lead délibérée et la Story 20.1 porte elle-même l'update du test.
