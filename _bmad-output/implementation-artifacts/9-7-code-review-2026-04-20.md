## Décision globale : **BLOCK**

L'implémentation des primitives (`common.py`) est **solide et de bonne qualité**. Mais trois findings **CRITICAL** invalident la valeur métier de la story :

1. **C1** — Les 34 tools de prod avalent leurs exceptions dans un `try/except Exception` et retournent `f"Erreur : {e}"` comme *valeur de retour normale*. Le wrapper `with_retry` **ne voit aucune exception transiente** → retry effectif = 0 %, circuit breaker effectif = 0 %, AC2/AC3/AC5/AC9 *sémantiquement non satisfaits* pour toute la surface métier.
2. **C2** — Les 27 tests "par module" (`_BaseModuleInstrumentation`) valident des **stubs locaux créés dans la classe de test** au lieu des tools de production. Aucun tool réel n'est couvert par les tests d'instrumentation. C'est cette décision (debug log T14) qui a masqué C1.
3. **C3** — AC4 "exhaustivité 12 colonnes" est vérifié **uniquement** via un `ac4_sample_tool` synthétique (lignes 291-323 de `test_tools_instrumentation.py`). Aucune validation que la journalisation fonctionne sur les vrais tools de prod qui passent par leur propre `try/except`.

Les 3 findings se renforcent : C2+C3 donnent une fausse assurance verte (1228 tests passent) sur un AC1/AC2/AC4/AC5 qui ne sont en réalité satisfaits que sur `interactive_tools.ask_interactive_question` et `guided_tour_tools.trigger_guided_tour` (+ les stubs de test).

**Chemin vers APPROVE** : appliquer les patches HIGH-1 (refactor exception handling) + HIGH-2 (tests contre tools prod) ci-dessous. Coût estimé : 2 à 4 h. Re-run tests : cible ≥ 1228, coverage ≥ actuelle.

## Findings

### CRITICAL

#### C1 — Les tools métier avalent les exceptions transientes, invalidant retry + circuit breaker
- **AC impactés** : AC2 (retry transient), AC3 (no-retry), AC5 (circuit breaker), AC8 (régression p95), AC9 (télémétrie `retry_recovered`)
- **Sévérité** : bloque merge
- **Évidence** (via `grep -n "except Exception" backend/app/graph/tools/*_tools.py`) :
  - `action_plan_tools.py` lignes 52 / 103 / 152
  - `application_tools.py` lignes 91 / 131 / 173 / 217 / 256 / 294
  - `carbon_tools.py` lignes 66 / 170 / 222 / 276
  - `chat_tools.py` lignes 74 / 124 / 180 / 230
  - `credit_tools.py` lignes 62 / 96 / 117
  - `document_tools.py` lignes 55 / 99 / 143
  - `esg_tools.py` lignes 46 / 133 / 193 / 258 / 353
  - `financing_tools.py` lignes 86 / 125 / 186 / 227
  - `profiling_tools.py` lignes 146 / 202

  Pattern type (`profiling_tools.py:146-148`) :
  ```python
  except Exception as e:
      logger.exception("Erreur dans update_company_profile")
      return f"Erreur lors de la mise à jour du profil : {e}"
  ```
  Conséquence : quand `asyncpg.exceptions.PostgresConnectionError` ou `httpx.ReadTimeout` survient dans `update_company_profile`, la fonction retourne une *string* au wrapper. Le wrapper interprète la string comme un succès (`status="success"`), incrémente `record_success()` du breaker, écrit une ligne `tool_call_logs.status="success"` — alors que métier a échoué.
- **Ce que l'on a vs ce qui est promis** :
  | Promis (AC2) | Livré |
  |---|---|
  | TimeoutError → 3 tentatives [1s, 3s, 9s] | 1 tentative, retour string "Erreur : …" |
  | 5xx LLM consécutifs → breaker ouvre à 10 | `_is_llm_5xx_error` reçoit 0 appel |
  | `retry_success` / `error` lignes avec `retry_count` | Une seule ligne `success`, log `tool_retry_recovered` jamais émis |
- **Fix (Option A — minimal, recommandé)** : modifier chaque `@tool` pour **ne plus avaler les exceptions transientes**. Deux patterns possibles :
  ```python
  # Pattern 1 : classifier dans le tool
  from app.graph.tools.common import is_transient_error
  try:
      ...
  except Exception as e:
      if is_transient_error(e):
          raise   # laisser le wrapper retry
      logger.exception("Erreur non-transient dans update_company_profile")
      return f"Erreur : {e}"

  # Pattern 2 : supprimer le try/except et laisser le wrapper gérer
  # (le wrapper a sa propre gestion error → return "Erreur : …")
  ```
  Le Pattern 2 est plus simple (34 sites de modification, ~100 LOC retirées). Le Pattern 1 préserve des messages utilisateur-friendly en cas d'erreur logique. **Décision D1 à trancher** (voir §Décisions).
- **Tests à ajouter** : après correction, un test par module qui lève `asyncio.TimeoutError` depuis le service sous-jacent (via monkeypatch de `company_service.update_profile` par exemple) et vérifie que 2 lignes `tool_call_logs` sont créées (`error` + `retry_success`). C'est la **vraie** validation AC2 sur tool de prod.

#### C2 — Tests d'instrumentation valident des stubs locaux au lieu des tools de prod
- **AC impactés** : AC1 (100 % wrapping), AC6 (coverage ≥ 85 %), AC7 (Epic 10 démarre instrumenté)
- **Sévérité** : bloque merge
- **Évidence** : `test_tools_instrumentation.py:154-241`
  ```python
  class _BaseModuleInstrumentation:
      def _build_success_tool(self):
          @tool_decorator
          async def test_success_tool(config: RunnableConfig = None) -> str:
              """Stub de succès."""
              return "OK"
          return with_retry(test_success_tool, max_retries=2, node_name=self.MODULE_NAME)
  ```
  Les 27 tests `TestXxxInstrumentation` (9 × 3) héritent de `_BaseModuleInstrumentation` et partagent tous **le même stub `test_success_tool`** — ils ne diffèrent que par `MODULE_NAME`. Aucun tool prod n'est invoqué.
- **Conséquence directe** : le test `test_retries_on_transient` est trivialement vert car `test_flaky_tool` lève `asyncio.TimeoutError` *hors* d'un try/except. Mais sur le vrai `update_company_profile`, la même exception serait avalée avant le wrapper. **Faux positif systématique**.
- **Raison donnée (debug log T14)** : « quand with_retry mute tool.coroutine en place, un monkeypatch tool.coroutine = fake court-circuite le wrapper ». Correct techniquement, mais le remède (stub local) évacue le risque que la story doit justement couvrir.
- **Fix** : remplacer les 27 tests par des tests qui :
  1. Importent le vrai tool (`from app.graph.tools.profiling_tools import update_company_profile` *avant* wrapping, ou extraire via `PROFILING_TOOLS[0]`).
  2. Monkeypatchent le **service sous-jacent** (ex. `app.modules.company.service.update_profile`) pour lever `asyncio.TimeoutError`.
  3. Invoquent `PROFILING_TOOLS[0].ainvoke({...}, config)`.
  4. Vérifient que 2 lignes `tool_call_logs` existent avec `status ∈ {error, retry_success}`.

  Cela valide simultanément C1 (après fix) et AC2 sur tool de prod.

#### C3 — AC4 "12 colonnes" vérifié uniquement sur stub synthétique
- **AC impactés** : AC4 (journalisation exhaustive)
- **Sévérité** : bloque merge (couplé à C1/C2)
- **Évidence** : `test_tools_instrumentation.py:288-323` — le seul test AC4 utilise `ac4_sample_tool` locale.
- **Fix** : ajouter 1 test AC4 par module (9 au minimum) invoquant le vrai tool via `XXX_TOOLS[0].ainvoke(...)` sur path heureux (mock du service métier pour retourner un résultat valide). Vérifier les 12 colonnes.

### HIGH

#### H1 — Doublon `create_fund_application` dans `FINANCING_TOOLS` et `APPLICATION_TOOLS` — nom conflictuel pour le LLM
- **AC impactés** : AC1 (registre unique), AC7 (Epic 10 — nom unique par tool pour ToolNode)
- **Évidence** : debug log story « 36 entrées — 34 distincts + doublon attendu `create_fund_application` présent dans financing ET application ». Dans `graph.py:140`, le noeud `application` reçoit `APPLICATION_TOOLS + INTERACTIVE_TOOLS`, et `financing` reçoit `FINANCING_TOOLS + …`. Le même `.name="create_fund_application"` dans 2 `ToolNode` différents est acceptable côté LangGraph, mais les 2 **fonctions implémentantes sont différentes** (body métier distinct). Le LLM, qui voit seulement `name` + docstring, peut invoquer la mauvaise implémentation selon le noeud actif.
- **Fix (Option A)** : renommer `application_tools.create_fund_application` en `submit_fund_application_draft` (sémantique métier claire). Préserve la sémantique, évite le conflit.
- **Fix (Option B)** : supprimer l'un des 2 et consolider sur l'autre. Préférer B si les 2 signatures métier sont identiques.
- **Si déféré** : créer story explicite dans `deferred-work.md` avec mention « duplicate tool name risk : LangGraph ToolNode ambiguity ».

#### H2 — `_safe_log` écrit dans la même session après une exception du tool métier → risque transaction avortée
- **AC impactés** : D11 (CCC-14 transaction boundaries), AC4 (log même sur erreur)
- **Évidence** : `common.py:520-530` (branche non-transient) appelle `_safe_log` *après* que `target_fn` a levé une exception. Si le tool métier a exécuté des `INSERT/UPDATE` avant l'exception (ex. `save_esg_criterion_score` échoue sur le 2ème commit après que le 1er est déjà flushed), la session SQLAlchemy est en état `PendingRollbackError` → `db.add(log_entry) + db.flush()` dans `log_tool_call` échouera. Le `try/except` dans `_safe_log` rattrape en DEBUG → on perd silencieusement la ligne de log d'erreur. Donc AC4 "journalisation exhaustive sur erreur" est partiellement tenu.
- **Fix** :
  1. Soit écrire les logs dans une session dédiée/savepoint (coût : refactor plus important, sort du scope 9.7).
  2. Soit documenter explicitement la limitation dans `README.md §5` et ajouter un test qui vérifie que le log est tenté (appel au moins observable via mock), même si persistence échoue.
  3. Option minimale : dans `_safe_log`, `logger.warning` au lieu de `logger.debug` quand le write échoue, pour ne pas manquer les pertes silencieuses.
- **Impact SRE** : sans ce log d'erreur, l'investigation AC qui a motivé la story (incident 2026-04-15) reste aveugle précisément sur les cas où le tool crashe en cours de transaction — ce sont justement les plus intéressants.

#### H3 — Double journalisation sur `interactive_tools.ask_interactive_question` (inline + wrapper)
- **AC impactés** : AC9 (télémétrie non-redondante), CQ-8 (pas de bruit dans `tool_call_logs`)
- **Évidence** : `interactive_tools.py:161-177` appelle `log_tool_call` inline en fin de corps tool, puis `INTERACTIVE_TOOLS = [with_retry(ask_interactive_question, max_retries=2, node_name="")]` ajoute un 2ème log via le wrapper. Chaque invocation crée 2 lignes identiques (ou quasi — sémantique enrichie côté inline).
- **Dev-agent l'a noté** dans findings déférés #2 (« acceptable car enrichissement sémantique préservé »). Désaccord review : la duplication alourdit les requêtes admin futures (FR55) et rend la p95 dashboard plus bruyante.
- **Fix** : supprimer l'appel inline dans `interactive_tools.py:161-177` (le wrapper couvre déjà AC4). Préserver le payload `tool_result` enrichi en renvoyant un résultat structuré que `_build_tool_result(result)` sérialisera. Même analyse pour `guided_tour_tools.py:98`.

#### H4 — Tests `test_logs_success` hériteront de TOUS les sous-ensembles via `_BaseModuleInstrumentation` — 9 tests strictement identiques
- **AC impactés** : AC6 (tests représentatifs, pas cosmétiques)
- **Évidence** : `TestProfilingInstrumentation(_BaseModuleInstrumentation)` ne surcharge que `MODULE_NAME = "profiling"`. Les 9 classes exécutent la même logique avec un node_name différent — ce qui n'apporte aucune garantie supplémentaire sur le module.
- **Fix** : chaque `TestXxxInstrumentation` devrait invoquer *au moins* un vrai tool de son module via `XXX_TOOLS[0]`, avec mock du service métier. Voir C2.

### MEDIUM

#### M1 — `_TOOL_ARGS_BLACKLIST` trop permissive (blacklist vs allowlist)
- **AC impactés** : AC4 (filtrage args), Sécurité : pas de PII loggée
- **Évidence** : `common.py:265-290`. Blacklist `{config, db, self}` seulement. Or plusieurs tools acceptent du texte libre :
  - `update_company_profile(governance_structure, environmental_practices, social_practices)` — 200 chars tronqués mais potentiellement info confidentielle.
  - `save_fund_interest(user_comment)`, `generate_application_section(section_content)` — idem.
- **Risque** : selon AC futures de conformité (RGPD, mention secrets client), `tool_args` JSON peut contenir des données qu'on ne veut pas persister dans `tool_call_logs`.
- **Fix** : ajouter dans `README.md §5` une contrainte explicite (« tout champ text libre > 50 chars doit être hashé ou non loggé ») + envisager une allowlist par tool. Déférer si jugé hors scope, mais créer une story explicite dans `deferred-work.md`.

#### M2 — `backoff=[]` avec `max_retries=0` accepté — opt-out silencieux du retry
- **AC impactés** : AC2 (intervalles stricts)
- **Évidence** : `common.py:402-406`. Un appel `with_retry(f, max_retries=0, backoff=[])` est accepté (0 < 0 est faux). C'est utilisé par `test_retry_e2e.py:146`. OK pour tests mais pattern exportable qui peut dégrader silencieusement le retry en prod si un dev copie-colle sans comprendre.
- **Fix** : documenter dans `README.md §3` que `max_retries=0` est un *opt-out* explicite et non une valeur par défaut. Ajouter `logger.info` au chargement du module si un tool opt-out.

#### M3 — `test_no_tool_escapes_wrapping` compare `declared - registered` mais pas `registered - declared`
- **AC impactés** : AC7 (protection anti-régression)
- **Évidence** : `test_tools_instrumentation.py:143-146`. Seule direction : si un `@tool` déclaré n'est pas enregistré → échec. L'inverse (tool enregistré mais plus déclaré) passe inaperçu.
- **Fix** : ajouter un assert inverse `assert not (registered_names - declared_tool_names)` et documenter les doublons acceptés (H1).

#### M4 — `_reset_breaker_state_for_tests()` bypasse le lock asyncio
- **AC impactés** : AC5 (thread-safety du breaker)
- **Évidence** : `common.py:254-257`. Clears synchrones `_counts.clear()` + `_opened_at.clear()` sans `async with self._lock`. Dans un run pytest-xdist concurrent, une fixture reset peut supprimer des counts pendant qu'un autre worker tient le lock pour `record_failure`.
- **Impact réel** : faible (tests en mémoire, faible parallélisme effectif sur le breaker), mais un test flaky futur pourrait en résulter.
- **Fix** : convertir `_reset_breaker_state_for_tests` en `async` ou utiliser `_breaker._counts = {}` via `_lock.locked()` check. Alternative : une fixture qui appelle `await _breaker.record_success(...)` sur toutes les clés actives.

## Décisions d'arbitrage à trancher

### D1 — Pattern correction du try/except dans les 34 tools (C1)
**Option A** : classifier avec `is_transient_error(e)` dans le tool et `raise` si transient. +messages métier préservés pour erreurs logiques, -200 LOC dupliquées × 34 tools.
**Option B** : supprimer le `try/except Exception` et laisser le wrapper gérer (il retourne déjà `f"Erreur : {e}"`). -34 × 3 LOC retirées, +wrapper centralisé, -messages utilisateur moins personnalisés.
**Recommandation review** : **Option B**. Le wrapper a déjà la bonne sémantique, et la diversité des messages d'erreur personnalisés n'apporte pas de valeur utilisateur réelle (le LLM reformule toujours).

### D2 — Périmètre tests contre tools prod (C2)
**Option A** : remplacer les 27 tests stubs par 27 tests prod (effort ~3 h).
**Option B** : ajouter 9 tests prod (1 par module) en gardant les 27 stubs comme smoke tests (effort ~1,5 h).
**Recommandation review** : **Option B**. Les stubs restent utiles pour le breaker et les checks de wiring ; 9 tests prod validés suffisent pour AC1/AC2/AC4 réels.

### D3 — Traitement H1 (doublon `create_fund_application`)
**Option A** : renommer dans `application_tools.py` vers `submit_fund_application_draft`.
**Option B** : consolider sur un seul tool en passant `draft=True|False` en paramètre.
**Recommandation review** : **Option A** à défaut d'analyse métier plus poussée. B serait idéal mais hors scope 9.7.
