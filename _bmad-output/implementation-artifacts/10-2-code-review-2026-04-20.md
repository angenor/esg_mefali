### [MEDIUM-10.2-1] `Company` modèle minimal (hors-scope AC2) — dédoublement sémantique avec `CompanyProfile`

**Fichiers** : `backend/app/modules/projects/models.py:51-69`, `backend/app/models/__init__.py:7,37`

Or, le repo contient déjà `app.models.company.CompanyProfile` (table `company_profiles`) qui représente le profil entreprise métier actuel. On se retrouve avec **deux modèles à sémantique voisine mais tables distinctes** :

| Modèle | Table | Usage | Story |

| `CompanyProfile` | `company_profiles` | Profil entreprise MVP (spec 003) | Livré |
| `Company` | `companies` | Socle Cluster A D1 Epic 11 | Squelette 10.2 |

**Risque** :
1. Un nouveau dev Epic 11 va importer le mauvais modèle par autocomplete (`from app.models.company import CompanyProfile` vs `from app.modules.projects.models import Company`).

**Recommandation** :

- **Option A (recommandée)** : documenter explicitement dans `deferred-work.md` la dette « **Migration Cluster A : CompanyProfile → Company** » sous Story 11-1 pour tracer que les deux coexistent jusqu'à migration complète, et ajouter dans la docstring de `projects/models.py::Company` une ligne `.. note:: Coexiste temporairement avec app.models.company.CompanyProfile (spec 003). Story 11-1 migrera les usages et dépréciera CompanyProfile en Story 20.1.`
- **Option B** (plus invasive) : dès Story 10.2, renommer le stub en `CompanyStub` ou `ProjectCompany` pour lever l'ambiguïté, quitte à renommer en `Company` dans Story 11-1.

**Décision souhaitée** : `A` (coût 5 min de doc, pas de refactor).

### [LOW-10.2-3] Test `flag OFF + no auth` non couvert

**Fichier** : `backend/tests/test_projects/test_router.py:58-68`

Le test 8 `test_endpoints_return_401_without_auth` définit explicitement `ENABLE_PROJECT_MODEL=true` pour ne pas masquer derrière 404. C'est correct. Mais **le cas `ENABLE_PROJECT_MODEL=false + no auth`** n'est pas couvert : on devrait vérifier que l'ordre `Depends(get_current_user)` AVANT `Depends(check_project_model_enabled)` donne 401 (pas 404) même quand le flag est OFF.

FastAPI résout les dependencies dans l'ordre de déclaration de la signature — donc le comportement attendu est préservé en théorie, mais un test défensif coûte 5 lignes.

**Patch** :

```python
@pytest.mark.asyncio
async def test_endpoints_return_401_without_auth_even_when_flag_disabled(
    client, monkeypatch
):
    """AC3 invariant : 401 prime sur 404 même quand le flag est OFF."""
    monkeypatch.setenv("ENABLE_PROJECT_MODEL", "false")
    for method, url, body in _ENDPOINTS:
        response = await _call(client, method, url, body)
        assert response.status_code == 401
```

## Layer 3 — Architecture & Traçabilité

### [MEDIUM-10.2-2] `_MODULE_ROUTE_FLAGS` absent de `"project"` → trap latent Epic 11

**Fichier** : `backend/app/graph/nodes.py:165-173,1315`

Le `project_node` retourne `active_module="project"` (AC6 §§3). Le dispatcher `router_node` (lignes 346-376) lit `active_module` et appelle `_is_topic_continuation(message, "project")` via **un appel LLM** (lignes 181-188 : `module_labels` n'inclut pas `"project"` → fallback au libellé brut `"project"`).

Puis :
```python
route_flag = _MODULE_ROUTE_FLAGS.get(active_module)  # → None pour "project"
if route_flag:
    route_flags[route_flag] = True  # skip — "project" pas dans le map
```

**Conséquences MVP** :
1. Si un état avec `active_module="project"` arrive au router (scénario interne / test qui invoque `project_node` puis ré-invoque `router_node` avec le state retourné), la classification LLM binaire est exécutée (coût token) alors qu'aucune route n'existe.
2. Le libellé affiché dans le prompt de classification est `"project"` brut (pas « module projet » fr) — dégrade la qualité de classification LLM.
3. Le node est unreachable depuis le router (AC6 §5 conforme), donc le trap ne se déclenche pas en production MVP.

**Mais** : quand Epic 11 ajoutera l'heuristique (via `_detect_project_request`), il **doit obligatoirement** :
- ajouter `"project": "_route_project"` dans `_MODULE_ROUTE_FLAGS`,
- ajouter `"project": "Module projet"` dans `module_labels` (nodes.py:181-189),
- ajouter `_route_project` dans `_route_after_router` (graph.py:19-38) ET dans le `conditional_edges` dict (graph.py:156-169).

Aucun test actuel ne vérifie cette cohérence. **Risque d'oubli latent** — exactement le type de trap que Story 9.7 C2 a voulu prévenir (scan anti-régression).

**Recommandations** (choisir 1) :

- **Option A (défensive, recommandée)** : ajouter un test `test_graph/test_module_route_flags_coherence.py` qui vérifie que **tout node dans `compiled_graph.nodes` avec un tool_loop a une entrée cohérente dans `_MODULE_ROUTE_FLAGS` OU est explicitement whitelisté comme "unreachable skeleton"**. Ajouter `project` à la whitelist.
- **Option B (minimale)** : ajouter un commentaire `TODO Epic 11 S1` explicite au-dessus de `_MODULE_ROUTE_FLAGS` (`# Epic 11 S1 : ajouter "project": "_route_project" avant d'activer le routing projet`) + ajouter le même TODO dans `module_labels`.
- **Option C** : tracer dans `deferred-work.md` sous Epic 11 Story 11-1 la checklist des 4 points à aligner.

**Décision souhaitée** : `B + C` (5 min, pas de nouveau test, mémo visible).

### [LOW-10.2-6] Double source of truth enum : migration 020 vs `enums.py`

**Fichiers** : `backend/alembic/versions/020_create_projects_schema.py:84-94` ↔ `backend/app/modules/projects/enums.py:12-19`

Les tests 3 et 4 (`test_project_status_enum_values`, `test_project_role_enum_values`) valident que **l'enum Python** a les bonnes valeurs, mais aucun test ne compare côte-à-côte migration ↔ enum. **Patch défensif** (ajout `test_models.py`) :

```python
def test_migration_enum_values_match_python_enum():
    """Les 5 valeurs du Python enum doivent matcher la migration 020."""
    # Les strings définies dans sa.Enum(...) migration 020
    migration_status = {"idea", "planning", "in_progress", "operational", "archived"}
    migration_role = {"porteur_principal", "beneficiaire", "partenaire", "observateur"}
    assert {m.value for m in ProjectStatusEnum} == migration_status
    assert {m.value for m in ProjectRoleEnum} == migration_role
```

Même si redondant avec tests 3 et 4, l'intention explicite (« migration ↔ enum synchronisés ») est précieuse comme garde-fou pour Story 11-1.
