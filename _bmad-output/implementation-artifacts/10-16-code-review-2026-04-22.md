## 2. Findings HIGH (4)

### H-1. Orphan `aria-describedby` → `hintId` non-rendu si `hint` + `error` cumulés (×3 composants)

- **Sources** : Blind Hunter
- **Sévérité** : HIGH (WCAG 4.1.2 Name, Role, Value)
- **Fichiers** :
  - `frontend/app/components/ui/Input.vue:63-68` + `:180-186`
  - `frontend/app/components/ui/Textarea.vue:56-62` + `:175-182`
  - `frontend/app/components/ui/Select.vue:52-57` + `:159-165`
- **Preuve** :
  ```ts
  const describedBy = computed(() => {
    const ids: string[] = [];
    if (props.hint) ids.push(hintId.value);     // toujours pushé si hint truthy
    if (props.error) ids.push(errorId.value);
    return ids.length ? ids.join(' ') : undefined;
  });
  ```
  ```vue
  <p v-if="hint && !error" :id="hintId">{{ hint }}</p>  <!-- hint masqué si error ! -->
  ```
- **Impact** : quand `hint` ET `error` sont passés simultanément, `aria-describedby="ui-input-xxx-hint ui-input-xxx-error"` référence un `hintId` dont l'élément n'est pas dans le DOM. NVDA/JAWS cherchent l'ID, ne le trouvent pas, annonce silencieusement raccourcie. Axe-core règle `aria-valid-attr-value` devrait le détecter — happy-dom le rate.
- **Patch** :
  ```ts
  const describedBy = computed(() => {
    const ids: string[] = [];
    if (props.hint && !props.error) ids.push(hintId.value);  // aligner avec v-if
    if (props.error) ids.push(errorId.value);
    if (props.showCounter) ids.push(counterId.value); // Textarea uniquement
    return ids.length ? ids.join(' ') : undefined;
  });
  ```
  + ajouter 1 test d'intégration par composant : monter avec `hint` + `error` et assert `aria-describedby` ne contient que l'errorId.

### H-2. Textarea `handleInput` mute `target.value` pendant composition IME (CJK + dead-keys accents FR)

- **Sources** : Blind Hunter + Edge Case Hunter (H-2)
- **Sévérité** : HIGH (accessibilité internationale)
- **Fichier** : `frontend/app/components/ui/Textarea.vue:98-107`
- **Preuve** :
  ```ts
  function handleInput(event: Event): void {
    const target = event.target as HTMLTextAreaElement;
    const truncated = target.value.slice(0, props.maxlength);
    if (truncated !== target.value) {
      target.value = truncated;  // ❌ mutation pendant composition IME
    }
    emit('update:modelValue', truncated);
  }
  ```
- **Impact** :
  - `@input` se déclenche **pendant** la composition IME sur Chromium (`isComposing === true`).
  - Pour utilisateurs CJK : glyphe partiel tronqué → IME avorte la saisie.
  - Pour claviers AZERTY avec dead-keys (é è ê ç — universels en Afrique francophone cible ESG Mefali) : un `a`+`◌̀` en composition peut être coupé si on est à 400 chars.
- **Patch** :
  ```ts
  function handleInput(event: Event): void {
    const ie = event as InputEvent;
    if (ie.isComposing) {
      emit('update:modelValue', (event.target as HTMLTextAreaElement).value);
      return; // pas de troncature pendant composition
    }
    // ... logique existante
  }
  // + @compositionend pour re-appliquer troncature une fois commit
  ```
  + test : `new InputEvent('input', { isComposing: true })` ne doit pas tronquer.
  + piège #17 codemap : « IME composition — ne pas muter pendant `isComposing` ».

### H-3. Select `:value="modelValue"` ne sélectionne pas un tableau sur `<select multiple>`

- **Sources** : Blind Hunter (H-1) + Edge Case Hunter (H-1 facette connexe)
- **Sévérité** : HIGH (fonctionnel — binding parent→enfant cassé)
- **Fichier** : `frontend/app/components/ui/Select.vue:102` + test `test_select_rendering.test.ts:101-104`
- **Preuve** :
  ```vue
  <select :value="modelValue" :multiple="multiple" ...>
  ```
  En mode `multiple=true` avec `modelValue = ['8', '13']`, le template utilise `v-bind` one-way (pas `v-model`). Vue applique `.value = ['8','13']` sur un `<select multiple>` natif → DOM sérialise en string `"8,13"`, aucune option ne matche → rien n'est sélectionné visuellement.
  
  Le test `AC3 : multiple=true → attr multiple + selectedOptions emit array` **contourne** le bug :
  ```ts
  Array.from(el.options).forEach((o) => {
    o.selected = o.value === '8' || o.value === '13';  // ❌ mutation imperative, jamais via modelValue
  });
  ```
- **Impact** : toute consommation de `<Select multiple v-model="ids" />` avec état initial non vide affiche un select vide. Story `ShowcaseGrid Multiple` avec `modelValue: ['8', '13']` visuellement cassée. Régression UX grave pour les cas ODD-cibles (ODD 8 + ODD 13 sont le pattern principal Spec 007 Carbon + 008 Financing).
- **Patch** : implémenter la sélection via `watch` sur `modelValue` + `selectedOptions` :
  ```ts
  import { watch } from 'vue';
  const selectRef = ref<HTMLSelectElement | null>(null);
  watch(
    () => props.modelValue,
    (val) => {
      if (!selectRef.value) return;
      if (props.multiple && Array.isArray(val)) {
        Array.from(selectRef.value.options).forEach((o) => {
          o.selected = val.includes(o.value);
        });
      }
      // single-select : :value binding natif fonctionne
    },
    { immediate: true },
  );
  ```
  + test : monter avec `modelValue: ['8', '13']` + `multiple: true`, assert `selectedOptions.length === 2`.
  + alternative plus propre : exposer le composant via `v-model` Vue natif (retirer le `handleChange` custom et laisser Vue gérer — mais cela contredit la signature `emit<'update:modelValue'>` explicite ; vérifier comportement Vue 3.5).

### H-4. Input `type="number"` émet string — contrat `modelValue: string | number` partiellement fictif

- **Sources** : Blind Hunter + Edge Case Hunter (H-4)
- **Sévérité** : HIGH (type contract + silent corruption)
- **Fichier** : `frontend/app/components/ui/Input.vue:27, 54, 107-112`
- **Preuve** :
  ```ts
  type InputProps = { modelValue: string | number; ... };
  const emit = defineEmits<{ 'update:modelValue': [value: string] }>();
  function handleInput(event: Event): void {
    const target = event.target as HTMLInputElement;
    emit('update:modelValue', target.value);  // ❌ toujours string
  }
  ```
  Le piège #11 codemap mentionne la coercion consommateur (`v-model.number`), mais :
  1. Aucun test ne verrouille le comportement string-only attendu.
  2. Consommateur passant `ref<number>(0)` sans `.number` reçoit silencieusement un `string`.
  3. Zod `z.number()` sur submit rejette — le bug ne se révèle qu'à la soumission.
- **Impact** : quantités FCFA (Spec 007 carbon, 008 financing) typées `number` en store Pinia corrompues silencieusement en `string`. Comparaisons `> 100000` → `"05" > 100000` = false via coercion lexicographique.
- **Patch (option 1, minimal)** : aligner le type prop avec l'emit + ajouter test
  ```ts
  modelValue: string; // retirer | number ; le consommateur coerce
  ```
  + test : `it('piège #11 : type=number émet toujours string', () => { expect(typeof emitted[0][0]).toBe('string'); })`
  + codemap §5 piège #11 : ajouter « le type prop `string | number` est une tolérance historique ; l'emit est toujours `string` ».
  
  **Patch (option 2, plus puissant)** : émettre `number` conditionnel à `type === 'number'` via `target.valueAsNumber` si fini. Contredit Q4 « primitive dumb » — **option 1 recommandée MVP**.

### M-1. Textarea `role="status" aria-live="polite"` toggle dynamique → annonce SR non fiable

- **Source** : Blind Hunter
- **Sévérité** : MEDIUM (a11y timing)
- **Fichier** : `Textarea.vue:188-189`
- **Preuve** :
  ```vue
  :role="isAtLimit ? 'status' : undefined"
  :aria-live="isAtLimit ? 'polite' : undefined"
  ```
- **Impact** : les régions `aria-live` doivent **exister dans le DOM avant** que le contenu change pour être annoncées. Toggler `role`/`aria-live` de `undefined` à `'status'/'polite'` au moment où on atteint 400 chars → NVDA/JAWS ratent l'annonce sur la première itération (région nouvellement créée).
- **Patch** : fixer `role="status" aria-live="polite"` en permanence sur le compteur :
  ```vue
  <p :id="counterId"
     :class="['text-xs tabular-nums flex-shrink-0', counterClasses]"
     role="status"
     aria-live="polite"
     aria-atomic="true">
    {{ currentLength }}/{{ maxlength }}
  </p>
  ```
  Éviter les annonces sur chaque frappe via `aria-live="polite"` (attend une pause) + `aria-atomic` pour annoncer la valeur complète. Tester avec NVDA dans Storybook CI (DEF-10.15-4).

### M-4. Input manque la défense troncature JS — asymétrie avec Textarea

- **Source** : Blind Hunter
- **Sévérité** : MEDIUM (cohérence triple défense Spec 018)
- **Fichier** : `Input.vue:107-112`
- **Preuve** : Textarea tronque côté JS (`target.value.slice(0, maxlength)`), Input ne fait qu'émettre `target.value` brut. Si un appelant fait `inputEl.value = 'x'.repeat(500); inputEl.dispatchEvent(new Event('input'))`, le HTML `maxlength` est bypass et Input émet 500 chars vers le modèle.
- **Impact** : Spec 018 `interactive_questions` justification utilise Textarea (OK). Mais champs libres (nom entreprise, adresse) via Input exposent un vecteur programmatique. Backend Pydantic valide — défense finale fonctionne — mais la parité avec Textarea manque.
- **Patch** : appliquer le même slice JS à Input quand `props.maxlength` est défini :
  ```ts
  function handleInput(event: Event): void {
    const target = event.target as HTMLInputElement;
    const value = props.maxlength != null
      ? target.value.slice(0, props.maxlength)
      : target.value;
    if (value !== target.value) target.value = value;
    emit('update:modelValue', value);
  }
  ```
  + test paste 500 chars dans Input `maxlength=100`.
  + mettre à jour piège #16 CODEMAPS pour explicitement couvrir Input (pas uniquement Textarea).
