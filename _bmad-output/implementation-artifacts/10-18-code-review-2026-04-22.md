### HIGH

#### HIGH-1 — AC6 bottom-sheet bascule mobile non implémentée (écart spec non documenté)

**Fichier** : `frontend/app/components/ui/Drawer.vue:61-74, 102`
**AC concerné** : AC6 (Mobile fullscreen automatique `<md` breakpoint)
**Sévérité** : HIGH (écart fonctionnel vs spec, pattern UX natif mobile
compromis)

**Constat** : AC6 prescrit explicitement :

> « l'animation slide-in **bascule sur `translate-y-full` (bottom-up)** au
> lieu de `translate-x-full` (right-slide) pour cohérence UX mobile native
> (pattern bottom-sheet) »

Le code livré applique uniformément `mobileBaseClasses = 'inset-0 w-full
h-full md:inset-auto'` quelle que soit la valeur de `side`. Sur mobile,
un `<Drawer side="right">` apparaît en plein écran sans distinction
visuelle de « panneau qui arrive par le bas » — pattern natif iOS
(UISheetPresentationController) et Android (BottomSheetDialog) perdu.

Pour `SourceCitationDrawer` (Epic 13, mobile PME persona dominante), la
consultation de sources RAG sur téléphone gagnerait significativement à
apparaître en bottom-sheet plutôt qu'en fullscreen neutre (affordance
« glisser pour fermer » implicite).

**Impact review** : cet écart n'est **pas listé** dans « Ajustements
mineurs vs spec » des Completion Notes (ligne 953-955), contrairement au
changement `<header>`/`<footer>` → `<div>`. Dette non traçable sans ce
rapport.

**Correctif suggéré** (choix orthogonal, coût ≤ 30 min) :

```diff
- const mobileBaseClasses = 'inset-0 w-full h-full md:inset-auto';
+ // Mobile : fullscreen pour side=right|left (pas de bottom-sheet
+ // distinct car viewport déjà plein) ; side=top|bottom conserve la
+ // sémantique verticale (top = descend du haut, bottom = monte du bas).
+ const mobileBaseClasses = computed<string>(() => {
+   if (props.side === 'bottom') {
+     return 'fixed bottom-0 left-0 right-0 w-full h-[90vh] md:inset-auto md:h-full';
+   }
+   if (props.side === 'top') {
+     return 'fixed top-0 left-0 right-0 w-full h-[90vh] md:inset-auto md:h-full';
+   }
+   return 'inset-0 w-full h-full md:inset-auto';
+ });
```

**Alternative** (si pattern natif iOS non souhaité Phase 0) :
explicitement documenter dans Completion Notes l'écart vs AC6 « spec
prescrivait bottom-sheet pattern, livré fullscreen neutre — DEF-10.18-X
pour Phase Growth ». La dette est **acceptable** si tracée — l'absence
de trace est le vrai problème.

**Tests manquants liés** : `test_drawer_behavior.test.ts` case
`mobile-fullscreen` (l.302-316) vérifie seulement les classes `w-full
h-full inset-0 md:inset-auto` présentes (pattern actuel). Aucun test
n'enforce `side=bottom → translate-y` vs `side=right → translate-x` sur
mobile. Régression silencieuse possible.

#### HIGH-2 — AC5/AC8 tests Escape et focus restoration permissifs

**Fichier** : `frontend/tests/components/ui/test_drawer_behavior.test.ts:163-184, 319-342`
**AC concernés** : AC5 (Escape ferme) + AC8 (focus restoration à la
fermeture)
**Sévérité** : HIGH (branche de test soft qui peut masquer régression
future)

**Constat 1 — test Escape permissif (AC5)** :

```ts
// l.180-182
if (events) {
  expect(events.at(-1)).toEqual([false]);
}
```

Ce pattern « teste seulement si l'event existe » transforme un test
**d'assertion** en test **de non-régression conditionnelle**. Si Reka UI
cesse d'émettre `update:open` sur Escape dans une version future (breaking
change majeur silencieux), le test reste vert car `events` serait
`undefined` et la branche est skipée. Le commentaire (l.178-180) justifie
cela par « happy-dom n'intercepte pas toujours le DialogRoot keydown »
mais c'est justement ce genre de divergence happy-dom/runtime qui doit
être **déléguée à Storybook runtime** (leçon 10.15 HIGH-2) plutôt que
faire porter au test unit un assert conditionnel.

**Correctif** : deux options mutuellement exclusives.

*Option A — forcer l'assert* :

```ts
const events = w.emitted('update:open');
expect(events).toBeDefined();
expect(events!.at(-1)).toEqual([false]);
```

Si ce test devient flaky en happy-dom, le **retirer** et capitaliser en
Storybook play function (cohérent délégation runtime).

*Option B — déplacer vers Storybook play function* : créer `CloseOnEscape`
play function avec `userEvent.keyboard('{Escape}')` + assert sur state
mutation `localOpen === false` via `canvas.findByRole('complementary')`
qui doit throw après close.

**Constat 2 — focus restoration AC8 non testé (AC8)** :

AC8 prescrit :

> « à la fermeture, focus restauré sur le déclencheur (gestion Reka UI
> natif via FocusTrap sous-jacent) »
>
> « un test case `focus-trap-disabled` vérifie que
> `document.activeElement` peut matcher un élément hors
> `[role="complementary"]` après Tab »
>
> « un test case `focus-trap-enabled` vérifie que `document.activeElement`
> reste dans le drawer après 3 Tab consécutifs. »

Les tests livrés l.320-341 :

```ts
it('drawer rendu sans erreur quand trapFocus=false (default)', …)
it('drawer rendu sans erreur quand trapFocus=true (opt-in)', …)
```

ne font qu'asserter `queryDrawer() !== null`. Le comportement focus réel
(Tab dans/hors drawer, restauration sur déclencheur) n'est pas couvert.
Les Completion Notes indiquent Tasks 5.2 cochée mais la vérification est
symbolique (« monte sans crash »), pas observable au sens Pattern A.

**Correctif** : basculer explicitement AC8 en délégation Storybook
runtime documentée (leçon 10.15 HIGH-2 déjà invoquée pour portail) et
retirer les 2 tests vidés de leur substance, OU ajouter des vrais asserts
`document.activeElement` (faisable en happy-dom avec
`focus-visible-polyfill` si pas déjà actif).

L'important : ne pas laisser un test vert qui prétend couvrir AC8
observables sans le faire. **Option privilégiée** : délégation Storybook
play function + note inline dans le test Vitest qui indique le délégat.

#### MEDIUM-2 — Runtime warn AC5 défense en profondeur strictement dev-only

**Fichier** : `frontend/app/components/ui/Drawer.vue:105-117`

```ts
if (import.meta.env.DEV) {
  if (!props.closeOnEscape && !props.closeOnOverlayClick && !props.showCloseButton) {
    console.warn('[ui/Drawer] 3 chemins fermeture desactives …');
  }

```

Le piège #32 (codemap §5) et AC5 dernier « And » prescrivent la défense
en profondeur. Limiter au mode DEV signifie qu'un consommateur Phase 1+

**Correctif** : soit retirer le garde `import.meta.env.DEV` (le
`console.warn` est bénin en prod, zéro impact perf, et protège contre le
user-trap réel), soit documenter explicitement la décision dans le
piège #32 codemap avec rationale « dev-only acceptable car test unit
catch en CI ». Actuellement, ni l'un ni l'autre.

**Note** : le test `test_drawer_behavior.test.ts:187-228` fonctionne car
Vitest tourne en mode DEV par défaut. OK pour CI. Mais la prod est un

#### MEDIUM-4 — AC13 test ScrollArea ne vérifie pas le data-attribute Reka UI

**Fichier** : `frontend/tests/components/ui/test_drawer_behavior.test.ts:344-362`
**AC concerné** : AC13 dernier « And »

  document.body.querySelector('.scroll-area-root');
expect(scrollRoot).not.toBeNull();

assertion structurelle via le nombre d'enfants DOM de `[role="complementary"]`
