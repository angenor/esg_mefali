### [HIGH-10.6-1] `delete_document` silencieusement no-op sur chemins legacy `uploads/<uid>/<did>/<file>` → fuite de fichiers disque

**Fichier** : `backend/app/modules/documents/service.py:633-641`
**AC référencé** : AC4.a — « Refactor `delete_document` (ligne 605-612) : remplace `_delete_file_from_disk(document.storage_path)` par `await storage.delete(document.storage_path)` » + invariant §1 Contexte « Les fichiers uploadés AVANT Story 10.6 restent accessibles via filesystem ; le path est stocké dans `Document.storage_path` qui continue d'être le **key** du provider ».

**Constat** : `delete_document` appelle directement `storage.delete(document.storage_path)` **sans** la rétrocompatibilité que `analyze_document` (ligne 384-395) applique. Conséquence concrète :

- **Nouveau document** (Story 10.6+) : `storage_path = "documents/<uid>/<did>/<file>"` → résolu par `LocalStorageProvider` en `backend/uploads/documents/<uid>/<did>/<file>` → fichier correctement supprimé. ✓
- **Document legacy** (avant Story 10.6) : `storage_path = "uploads/<uid>/<did>/<file>"` (pattern `_save_file_to_disk` originel qui préfixait avec `uploads/`) → `LocalStorageProvider.delete` résout `base_path / key = backend/uploads / uploads/<uid>/<did>/<file> = backend/uploads/uploads/<uid>/<did>/<file>`. Ce fichier n'existe **pas** (double `uploads/`) → `unlink(missing_ok=True)` ne lève rien → le fichier réel à `backend/uploads/<uid>/<did>/<file>` **reste orphelin sur disque**. ✗

**Risque opérationnel** :

1. **RGPD / droit à l'effacement** : un utilisateur qui supprime son compte pensera avoir effacé ses PDF sensibles (SGES, bilan financier, RCCM). Le fichier physique persiste, violant NFR46 et le principe de minimisation.
2. **Saturation disque silencieuse** : les anciens uploads s'accumulent sans nettoyage. NFR69 budget storage biaisé.
3. **Bascule Phase Growth cassée** : le script `migrate_local_to_s3.py` déferré (deferred-work §Storage) copiera ces orphelins vers S3, créant une divergence BDD↔S3 permanente.

**Preuve factuelle** : `analyze_document` (lignes 383-390) contient la logique rétrocompat suivante :

```python
if isinstance(storage, LocalStorageProvider):
    if document.storage_path.startswith("uploads/"):
        file_path = str(UPLOADS_DIR.parent / document.storage_path)  # ← legacy
    else:
        file_path = str(storage.local_path(document.storage_path))   # ← nouveau
```

Cette logique est **absente** de `delete_document:637-639` :

```python
async def delete_document(db: AsyncSession, document: Document) -> None:
    storage = get_storage_provider()
    await storage.delete(document.storage_path)  # ← pas de rétrocompat
    await db.delete(document)
    await db.flush()
```

Grep confirme :

```bash
rg -n "storage_path.startswith" backend/app/modules/documents/
# service.py:387 (analyze_document seulement) — delete_document non couvert
```

Aucun test `test_documents_*` ne couvre la chaîne « upload style legacy → delete → fichier physique absent » (les tests utilisent la fixture auto-use `isolate_storage_provider` qui part d'un répertoire vide, donc les paths legacy ne sont jamais produits).

**Recommandation (choisir 1)** :

- **Option A (recommandée, ~15 lignes)** — extraire la rétrocompat dans un helper partagé :

  ```python
  # backend/app/modules/documents/service.py
  def _resolve_legacy_storage_path(storage_path: str) -> str | None:
      """Retourne un path filesystem absolu si `storage_path` est un legacy
      (`uploads/<uid>/<did>/<file>`), sinon None. Story 10.6 rétrocompat."""
      if storage_path.startswith("uploads/"):
          return str(UPLOADS_DIR.parent / storage_path)
      return None

  async def delete_document(db: AsyncSession, document: Document) -> None:
      legacy_path = _resolve_legacy_storage_path(document.storage_path)
      if legacy_path is not None:
          # Legacy : unlink direct (bypass provider qui a base_path différent)
          Path(legacy_path).unlink(missing_ok=True)
      else:
          storage = get_storage_provider()
          await storage.delete(document.storage_path)
      await db.delete(document)
      await db.flush()
  ```

  + test paramétré `test_delete_document_legacy_and_new_paths` qui seed un fichier à `backend/uploads/<uid>/<did>/legacy.pdf` via `tmp_path`, vérifie qu'il disparaît après `delete_document`.

- **Option B** — migration forcée au boot : scan une fois `Document.storage_path` et réécrit les rows legacy via un script Alembic data-migration. Plus invasif, mais élimine la dette rétrocompat.

- **Option C (non recommandée)** — accepter la dette avec un `deferred-work` explicite + issue bloquante Phase Growth (avant `migrate_local_to_s3.py`). **Risque RGPD non neutralisé** → pas acceptable seul.

**Référence croisée** : la même classe d'asymétrie concerne `reports/router.py:102-106` (rétrocompat legacy `file_path` sans préfixe `reports/`) qui **est correctement implémentée** — incohérence à aligner.

- Phase Growth : Story 10.7 active CRR `eu-west-3 → eu-west-1` ; si le bucket source est dans une autre région, la CRR silencieusement inactive → NFR33 backup 2 AZ cassée.

### [MEDIUM-10.6-3] SSE-S3 AES256 prouvé uniquement sur `put_object` (single-part) — **pas** sur `upload_fileobj` (multipart ≥ 10 MB)

**Fichier** : `backend/tests/test_core/test_storage/test_s3_provider.py:61-66` + `test_s3_error_paths.py:89-97`
**AC référencé** : AC6.3 — « `test_put_sse_applied` — head_object retourne `ServerSideEncryption="AES256"` (NFR25 prouvé end-to-end) ».

**Constat** : `test_put_sse_applied` utilise `b"secret-data"` (11 bytes) → branche `len(content) < MULTIPART_THRESHOLD_BYTES` → chemin `put_object` direct avec `ServerSideEncryption="AES256"`. Test vert. ✓

Mais la branche multipart (`upload_fileobj` pour `bytes ≥ 10 MB` ou `BinaryIO` en sans taille), qui passe SSE via `ExtraArgs={"ServerSideEncryption": "AES256", ...}` ligne `s3.py:159`, **n'est pas vérifiée** :

- `test_put_multipart_via_binaryio` (test_s3_error_paths.py:89) fait un round-trip 2 MB via BinaryIO mais assert uniquement sur la longueur du contenu, **pas sur `head_object.ServerSideEncryption`**.
- `test_roundtrip_s3` (test_providers_e2e.py:52) fait 10 MB via `bytes` → basculé en multipart + assert sur `len(got) == size_mb * 1024 * 1024`, **pas sur SSE**.

**Risque** : si un futur refactor supprime `"ServerSideEncryption": "AES256"` du dict `extra_args` (s3.py:159) mais conserve la branche single-part, la suite reste **verte** et NFR25 est silencieusement violée sur les gros uploads (typiquement les bilans PDF 15-50 MB qui sont précisément ceux à chiffrer).

**Limitation connue moto** : moto ≥ 5.0 propage ExtraArgs dans la plupart des cas, mais le test doit le vérifier explicitement pour immuniser le code contre une régression.

**Recommandation (~10 lignes)** :

Ajouter dans `test_s3_provider.py` ou `test_s3_error_paths.py` :

```python
async def test_put_sse_applied_on_multipart_binaryio(s3_provider: S3StorageProvider):
    """NFR25 — SSE-S3 AES256 doit s'appliquer aussi sur le chemin upload_fileobj."""
    buffer = BytesIO(b"X" * (2 * 1024 * 1024))  # 2 MB via BinaryIO → multipart
    await s3_provider.put("multipart/sse.bin", buffer)
    client = boto3.client("s3", region_name=REGION)
    head = client.head_object(Bucket=BUCKET, Key="multipart/sse.bin")
    assert head.get("ServerSideEncryption") == "AES256"


async def test_put_sse_applied_on_large_bytes(s3_provider: S3StorageProvider):
    """Bytes ≥ 10 MB basculent en multipart — SSE doit suivre."""
    payload = b"Y" * (10 * 1024 * 1024 + 1)  # juste au-dessus du seuil
    await s3_provider.put("multipart/large.bin", payload)
    client = boto3.client("s3", region_name=REGION)
    head = client.head_object(Bucket=BUCKET, Key="multipart/large.bin")
    assert head.get("ServerSideEncryption") == "AES256"
```

**Scope** : 2 tests × ~8 lignes = immunisation NFR25 sur 100 % du chemin put.

### [MEDIUM-10.6-4] Contrat `StorageProvider.signed_url` : sémantique divergente entre Local (raise NotFound) et S3 (émet URL valide même absente)

**Fichier** : `backend/app/core/storage/base.py:68-72` vs `local.py:178-185` vs `s3.py:263-281`
**AC référencé** : AC1 — docstring ABC « Lève `StorageNotFoundError` si absent (local). TTL borné [1, 3600] côté implémentations S3. » — mention locale entre parenthèses = **contrat ambigu**.

**Constat** : les deux implémentations ont des sémantiques différentes sur clé absente :

| Provider | Clé absente | Comportement |
|---|---|---|
| `LocalStorageProvider` | `signed_url(missing)` | Lève `StorageNotFoundError` (local.py:181-182) |
| `S3StorageProvider` | `signed_url(missing)` | Émet URL pré-signée valide (boto3 ne fait pas de round-trip serveur), l'accès renverra 404 côté consommateur |

La Story documente ce choix (AC3 « Pas de pré-check `exists` avant (round-trip réseau inutile) ») mais l'ABC `base.py:68-72` indique « Lève `StorageNotFoundError` si absent (local) » — le « (local) » est une note entre parenthèses qui **ne figure pas comme invariant** dans le contrat de 6 méthodes. Les tests existants valident chaque provider séparément :

- `test_signed_url_missing_raises_not_found` (test_local_provider.py:75) ✓ pour local
- pas de test équivalent pour S3 (car comportement voulu)

**Risque** : les consommateurs qui codent contre l'abstraction `StorageProvider` (pattern Factory injection) écriront :

```python
try:
    url = await storage.signed_url(key)
    return RedirectResponse(url)
except StorageNotFoundError:
    raise HTTPException(404)
```

En mode Local, 404 correct. En mode S3, **pas de 404 côté backend** → le client reçoit un 302 + URL signée qui retournera 404 S3 → UX incohérente + signal de sécurité flou (l'attaquant apprend que la clé n'existe pas *après* avoir reçu une URL signée, ce qui peut être exploité en énumération temporelle).

**Recommandation (choisir 1)** :

- **Option A (recommandée, ~3 lignes)** — clarifier le contrat ABC et la storage.md §2 :

  ```python
  # backend/app/core/storage/base.py
  @abstractmethod
  async def signed_url(self, key: str, *, ttl_seconds: int = 900) -> str:
      """Retourne une URL pré-signée (ou URI `file://` pour local).
      TTL borné [1, 3600] côté implémentations S3.

      **Contrat de présence** (divergent) :
        - Local : lève `StorageNotFoundError` si absent (check synchrone).
        - S3 : émet l'URL sans pré-check (économise 1 round-trip HEAD) —
          le 404 n'apparaîtra qu'à la consommation de l'URL par le client.

      Les consommateurs qui requièrent une garantie de présence doivent
      appeler `await storage.exists(key)` avant `signed_url(key)`.
      """
  ```

  + mettre à jour `docs/CODEMAPS/storage.md §2` pour documenter cette divergence explicitement.

- **Option B** — harmoniser : ajouter un `head_object` pré-check dans `S3StorageProvider.signed_url`. Coût : 1 round-trip réseau par call + +5 ms latence. Va à l'encontre de la justification AC3.

Préférence : **Option A** (documentation contractuelle) car la performance prime et le pattern « check then sign » est déjà appliqué côté `reports/router.py:120` via `try/except Exception`. Le consommateur qui veut la garantie doit faire `exists()` explicite.

### [LOW-10.6-5] Filename en clair dans la clé opaque → PII potentielle en logs/traces IAM/CloudTrail

**Fichier** : `backend/app/core/storage/keys.py:30`
**AC référencé** : AC2 + storage.md §3 — « Structure `<base>/documents/<uid>/<did>/<filename>` ».

**Constat** : `storage_key_for_document(user_id, document_id, filename)` produit une clé incluant le `filename` utilisateur sanitisé :

```python
return f"documents/{user_id}/{document_id}/{filename}"
```

Le filename est déjà sanitisé (`_sanitize_filename` ligne 80-92 documents/service.py retire `..`, `/`, `<>:\"'|?*`) mais conserve le nom "humain" (ex. `"contrat_mhamadou_sgbs_2026.pdf"`).

**Risque** :

1. **CloudTrail logs S3** enregistrent `EventName=GetObject` + `RequestParameters.key=documents/<uid>/<did>/contrat_mhamadou_sgbs_2026.pdf` → **PII (nom client) exposée aux admins AWS**.
2. **IAM policy debugging** : si un SRE active `s3:Get*` audit via Access Analyzer, les filenames apparaissent dans les résultats.
3. **Retry logs applicatifs** : `s3.py:137` logge `"s3.%s transient error"` avec `exc` → le message ClientError peut contenir la clé complète (filename inclus) → logs FastAPI → stockage logs (typiquement 90 jours).
4. **Cache HTTP / ETag** : un proxy qui bufferise les signed URLs verra le filename dans l'URL signée.

**Exploitabilité** : FAIBLE car (a) CloudTrail est restreint aux admins Mefali internes (whitelist ADMIN_SUPER_EMAILS), (b) les filenames sensibles sont une minorité en MVP. Mais **capitalisation prévisible** : en Phase Growth, quand la volumétrie grandit et quand les bailleurs exigent des audits, cette couverture log devient un point de friction compliance.

**Recommandation (Phase Growth, ~5 lignes)** :

Ajouter un mode d'opacification du filename dans la clé :

```python
# backend/app/core/storage/keys.py
import hashlib

def storage_key_for_document(
    user_id: uuid.UUID,
    document_id: uuid.UUID,
    filename: str,
    *,
    opaque_filename: bool = False,  # Phase Growth : True par défaut
) -> str:
    if opaque_filename:
        # SHA256[:16] + extension originale
        digest = hashlib.sha256(filename.encode()).hexdigest()[:16]
        ext = Path(filename).suffix
        leaf = f"{digest}{ext}"
    else:
        leaf = filename
    return f"documents/{user_id}/{document_id}/{leaf}"
```

Le `Document` ORM stocke le filename original en `original_filename` pour affichage UI (déjà fait) → aucune perte UX. Tracé dans `deferred-work.md` sous **« Opacification filename en clé Phase Growth »**.

### [LOW-10.6-6] IAM `s3:DeleteObject` accordé à l'application → risque suppression de masse si RLS compromis

**Fichier** : `docs/CODEMAPS/storage.md:73-89` (IAM policy minimale documentée)

**Constat** : la policy documentée grant à l'application 3 actions de mutation :

```json
{
  "Action": ["s3:GetObject", "s3:PutObject", "s3:DeleteObject"],
  "Resource": "arn:aws:s3:::mefali-uploads-prod/*"
}
```

Si un attaquant compromet le pod backend (RCE via CVE LangChain/FastAPI) OU bypasse la RLS Story 10.5 via un admin malveillant OU via SQL injection **dans une query vectorielle** non paramétrée (NFR66 scan à refaire régulièrement) → il peut appeler `storage.delete(key)` en boucle sur toutes les clés listées par `list_objects_v2` → **suppression irréversible de tous les uploads clients**.

**Risque** : destructeur à l'échelle. Récupération via CRR EU-West-3 → EU-West-1 (Story 10.7) **uniquement si** les suppressions sont répliquées avec `DeleteMarker` + bucket versioning activé (actuellement non documenté — cf. LOW-10.6-8).

**Recommandation (Phase Growth)** :

- Séparer les rôles IAM : l'app backend n'a que `s3:GetObject` + `s3:PutObject` ; les deletes passent par un job admin dédié (Cron + rôle séparé avec MFA delete).
- Alternative légère : scoper le `s3:DeleteObject` au préfixe `"<bucket>/temp/*"` uniquement (pour cleanup caches), et les suppressions "utilisateur" passent par un soft-delete (flag `Document.deleted_at`) + purge offline.
- À tracer dans `deferred-work.md` sous **« Durcissement IAM S3 Phase Growth »**.

### [LOW-10.6-7] `LocalStorageProvider.exists()` avale silencieusement `StoragePermissionError` sur tentative path-traversal

**Fichier** : `backend/app/core/storage/local.py:168-174`

**Constat** :

```python
async def exists(self, key: str) -> bool:
    try:
        _validate_key(key)
    except StoragePermissionError:
        return False
    dest = self._base_path / key
    return await asyncio.to_thread(dest.is_file)
```

Le contrat ABC dit « Retourne True si `key` existe, False sinon (jamais d'exception) ». L'implémentation respecte le contrat mais **masque un signal d'attaque** : si un appelant passe `"../../../etc/passwd"`, `exists()` retourne `False` comme si le fichier n'existait pas, au lieu de logger/alerter.

**Risque** : FAIBLE (défense en profondeur — les clés sont produites par `storage_key_for_*` sûr). Mais un futur endpoint qui expose `exists(user_controlled_key)` serait vulnérable à l'énumération silencieuse.

**Recommandation (~2 lignes)** :

```python
async def exists(self, key: str) -> bool:
    try:
        _validate_key(key)
    except StoragePermissionError as exc:
        logger.warning("exists() on suspicious key: %s (%s)", key, exc)
        return False
    ...
```

**Fichier** : `backend/app/core/storage/s3.py:41-50`

**Constat** :

```python
_TRANSIENT_ERROR_CODES = frozenset({
    "RequestTimeout", "RequestTimeoutException",
    "ServiceUnavailable", "SlowDown", "InternalError",
    "ThrottlingException",
})
```

`ThrottlingException` est canonique pour DynamoDB/Lambda/API Gateway. S3 utilise `SlowDown` (déjà présent) et `503 SlowDown` pour throttling. `ThrottlingException` sur S3 n'est documenté nulle part chez AWS — inclusion défensive mais inutile.

**Impact** : négligeable (code mort). Mais peut induire en erreur lors d'un audit du retry policy.

**Recommandation** : retirer `ThrottlingException` OU renommer la constante `_TRANSIENT_AWS_ERROR_CODES` pour indiquer qu'elle est volontairement pan-AWS (si le module devait servir d'autres services AWS un jour).

### [LOW-10.6-10] `S3StorageProvider.signed_url` délègue `generate_presigned_url` à `asyncio.to_thread` — overhead inutile

**Fichier** : `backend/app/core/storage/s3.py:269-275`

**Constat** : `generate_presigned_url` est **purement local** (signature HMAC, aucun round-trip réseau). Le délégué au thread pool ajoute ~0.3-0.8 ms par call vs exécution inline. Le commentaire inline (lignes 266-268) reconnaît ce point :

```python
# generate_presigned_url est sync mais purement local (signature HMAC)
# → pas de round-trip réseau, mais on délègue au thread pool pour
# cohérence et éviter tout blocage sur crypto lente.
```

« éviter tout blocage sur crypto lente » est correct pour les signatures RSA (SSH) mais **pas pour HMAC-SHA256** qui fait ~50 μs. Le thread-pool ajoute plus d'overhead qu'il ne sauve.

**Impact** : 0.5-1 ms × 100 req/s download = +0.05-0.1 vCPU permanent en Phase Growth. Négligeable MVP.

**Recommandation** : documenter en INFO et laisser tel quel (cohérence > micro-optim). Flag à revisiter si profiling Phase Growth montre saturation du thread pool.

## Layer 3 — Architecture & Leçons

### [INFO-10.6-11] Règle d'or 10.5 (E2E effet observable) respectée avec 2 exceptions justifiées

Les 49 tests suivent le pattern « put → get → assert binary equality » sans mock des internals. Deux tests font un `monkeypatch` ciblé, **tous deux justifiés** car l'effet à observer n'est pas reproductible naturellement via moto :

1. `test_retry_on_transient_then_success` (test_s3_error_paths.py:110) — monkeypatch sur `_client.put_object` pour simuler 1 SlowDown puis succès. **Justifié** : moto n'émet pas de SlowDown sans coop explicite. Le test vérifie l'effet observable = `call_count == 2` + upload final réussi.
2. `test_exists_on_access_denied_raises` (test_s3_error_paths.py:166) — monkeypatch `_client.head_object` pour lever AccessDenied. **Justifié** : l'ACL n'est pas trivialement forçable sous moto sans bootstraper un bucket policy complet.

Ces exceptions sont **alignées** avec la leçon 9.7 C2 (tests prod véritables) : on mocke le minimum nécessaire pour déclencher la branche visée, pas la bibliothèque entière.

Pattern contraire (anti-leçon, **absent ici**) : `patch("boto3.client")` qui retournerait un `MagicMock` générique. Aucun test 10.6 ne le fait — confirmé par `rg -n "patch.\".*boto3" backend/tests/test_core/` = 0 match.

### [INFO-10.6-12] Factory `get_storage_provider` + `lru_cache` cohérente avec Story 10.5 `rls.py` — pattern réutilisable Epic 11+

**Fichier** : `backend/app/core/storage/__init__.py:45-78` vs `backend/app/core/rls.py` (référence)

Le pattern suivant est désormais **standardisé** dans `core/` :

1. Factory `@lru_cache(maxsize=1)` sync qui retourne un singleton process-level.
2. Reset helper privé `_reset_<X>_cache()` non exporté dans `__all__` mais importable via `app.core.X._reset_...` pour les fixtures de test.
3. Lecture tardive `from app.core.config import settings` **à l'intérieur** de la factory (évite les imports circulaires au boot).
4. Fixture auto-use `isolate_<X>` dans `tests/conftest.py` qui patch `settings` + `_reset_<X>_cache()` avant/après chaque test.

Ce pattern est **reproductible tel quel** pour Epic 11 (micro-Outbox `get_outbox_dispatcher`), Epic 12 (formalization calculator `get_formalization_calculator`), et Epic 17 (LLM provider switch D10 `get_llm_provider`).

**Recommandation** : capitaliser dans un document d'architecture transverse (`docs/CODEMAPS/core-patterns.md` ou section existante) pour que les implémenteurs Epic 11+ ne réinventent pas le pattern. Pas bloquant pour merge.
