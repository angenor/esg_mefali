#### HIGH-10.8-2 — Scan AC4 ignore la duplication migrée vers `graph/nodes.py` et `prompts/system.py`

**Fichier** : `backend/tests/test_prompts/test_no_duplicate_imports.py:18-26`

**Symptôme** : `test_no_duplicate_instruction_imports_in_business_modules` itère sur `PROMPTS_DIR.glob("*.py")` et exclut 4 fichiers (`registry.py`, `system.py`, `widget.py`, `guided_tour.py`). La portée est **strictement limitée à `backend/app/prompts/`**. Mais une recherche backend complète (`grep -rn "WIDGET_INSTRUCTION|STYLE_INSTRUCTION|GUIDED_TOUR_INSTRUCTION" backend/app/`) révèle :

```
app/graph/nodes.py:1195: from app.prompts.widget import WIDGET_INSTRUCTION
app/graph/nodes.py:1197: full_prompt = system_prompt + tool_instructions + "\n\n" + WIDGET_INSTRUCTION
app/prompts/system.py:214: sections.append(STYLE_INSTRUCTION)         # exempt par ALLOWED_FILES
app/prompts/system.py:222: GUIDED_TOUR_INSTRUCTION,                    # exempt
app/prompts/system.py:225: sections.append(GUIDED_TOUR_INSTRUCTION)    # exempt
```

L'invariant CCC-9 « zéro concaténation manuelle des 3 constantes transverses dans le code applicatif » devrait s'appliquer **à tout le backend**, pas seulement à `prompts/`. Sinon le test passe vert pendant que la duplication migre silencieusement vers d'autres dossiers (cas réel : `nodes.py:1197`).

**Conséquence** : régression future invisible. Si un dev refactore demain `esg_node` pour ajouter `+ WIDGET_INSTRUCTION` directement dans `nodes.py`, le test passe toujours.

**Remédiation** :
```python
# test_no_duplicate_imports.py — élargir la portée:
BACKEND_APP = BACKEND_ROOT / "app"
ALLOWED_PATHS = {
    BACKEND_APP / "prompts" / "registry.py",
    BACKEND_APP / "prompts" / "system.py",     # bypass documenté Dev Notes §5
    BACKEND_APP / "prompts" / "widget.py",
    BACKEND_APP / "prompts" / "guided_tour.py",
}

@pytest.mark.unit
def test_no_duplicate_instruction_imports_anywhere() -> None:
    offenders: dict[str, list[int]] = {}
    for py_file in BACKEND_APP.rglob("*.py"):
        if py_file in ALLOWED_PATHS:
            continue
        content = py_file.read_text(encoding="utf-8")
        matches = [
            ln for ln, line in enumerate(content.splitlines(), 1)
            if FORBIDDEN_PATTERN.search(line)
        ]
        if matches:
            offenders[str(py_file.relative_to(BACKEND_APP))] = matches
    assert not offenders, (
        f"Duplication CCC-9 résiduelle hors prompts/ : {offenders}"
    )
```

À noter : ce test va **immédiatement échouer** sur `nodes.py:1195`, ce qui force la résolution de HIGH-10.8-1. Les deux findings sont liés.

#### MEDIUM-10.8-4 — `build_prompt` ne pré-valide pas l'agrégat des `required_vars` manquantes (multiples passages)

**Fichier** : `backend/app/prompts/registry.py:161-177`

**Symptôme** : la boucle `for entry in applicable:` lève `UnboundPromptVariableError` à la **première** entrée incomplète. Si trois entrées requièrent chacune une variable manquante, l'utilisateur doit corriger une à une au lieu d'avoir le diagnostic complet d'un coup.

**Conséquence** : DX dégradée pour le futur (pas critique en MVP avec `required_vars=()`).

**Remédiation** : pré-validation en 2 passes (collecte agrégée puis substitution) :
```python
applicable = sorted(...)

# Pré-validation agrégée (toutes les variables manquantes en un coup)
all_missing: dict[str, list[str]] = {}
for entry in applicable:
    missing = [v for v in entry.required_vars if v not in vars_map]
    if missing:
        all_missing[entry.name] = missing
if all_missing:
    raise UnboundPromptVariableError(
        f"Variables manquantes pour module {module!r} : {all_missing}"
    )

# Substitution
rendered: list[str] = []
for entry in applicable:
    try:
        rendered.append(Template(entry.template).substitute(vars_map))
    except KeyError as exc:
        raise UnboundPromptVariableError(...) from exc
```

À traiter ou défer selon priorité — pas critique aujourd'hui car `required_vars` est vide pour les 3 instructions actuelles.

### ℹ️ INFO

#### INFO-10.8-1 — Pattern « commit golden intermédiaire pré-refactor » à capitaliser dans BMAD process

Le commit `38e6d0f chore(10.8): freeze golden snapshots pré-refactor` séparé du refactor `c546b49` est un excellent pattern reproductible pour Epic 11+ (refactor metier). Avantages :
1. **Bisect-friendly** : si une régression sémantique apparaît, `git bisect` isolera précisément le commit fautif (pas mélangé avec la création des goldens).
2. **Revue séparée** : le commit golden est purement « capture d'état » (review = vérifier que les goldens représentent bien le comportement actuel), distinct du commit refactor (review = vérifier que la transformation préserve le comportement).
3. **Rollback ciblé** : `git revert c546b49` restaure le code avant refactor sans toucher aux goldens, qui restent comme baseline pour une 2ᵉ tentative.

**Recommandation** : ajouter ce pattern à `bmad-dev-story` workflow (post-Task 1 « Scan NFR66 ») comme **Task standardisée** « Capture goldens pré-refactor (commit isolé) » pour toute story de refactor metier qui touche un comportement observable (prompts, formats de sortie, calculs).
