### 🔴 HIGH

#### HIGH-10.7-1 — Workflow `schedule: cron` anonymisation PROD→STAGING absent

**Fichiers** : `.github/workflows/` (aucun workflow d'anonymisation), `docs/runbooks/README.md:234`
**Symptôme** : le runbook §7 Refresh STAGING déclare : « **Programmé** : 1er du mois, 02:00 UTC — GitHub Actions `schedule: cron '0 2 1 * *'` (Q4 tranchée Story 10.7, EventBridge différé Phase Growth) ». La décision architecture D8 parle de « copie mensuelle automatisée + validation fail-fast ». Or **aucun** des 3 workflows livrés ne contient de trigger `schedule:`, et il n'existe pas de `anonymize-refresh.yml` dédié.

**Conséquence** :
- Le CLI `anonymize_prod_to_staging.py` existe et est testé, mais n'est **jamais déclenché automatiquement**.
- Refresh STAGING = procédure 100 % manuelle (runbook étape 5 : opérateur exécute `python -m scripts.anonymize_prod_to_staging`).
- Drift STAGING vs PROD garanti (NFR73 isolation respectée, mais utilité STAGING dégradée → tests intégration sur data stale).
- **La story réclame 9 AC livrés** ; AC5 livre le script mais pas l'**orchestration** — la story a 8.5 AC livrés.

**Remédiation** :
Créer `.github/workflows/anonymize-refresh.yml` :
```yaml
name: Refresh STAGING (monthly anonymized copy)
on:
  schedule:
    - cron: '0 2 1 * *'  # 1er du mois 02:00 UTC
  workflow_dispatch:
permissions:
  id-token: write
  contents: read
jobs:
  refresh:
    runs-on: ubuntu-latest
    environment:
      name: staging  # Gate approval même en cron (audit trail)
    steps:
      - uses: actions/checkout@v4
      - name: Configure AWS (staging role)
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: ${{ secrets.AWS_ROLE_TO_ASSUME_STAGING }}
          aws-region: eu-west-3
      - name: Dump PROD (filtered)
        run: |
          aws rds create-db-snapshot --db-snapshot-identifier refresh-$(date +%Y%m%d) ...
      - name: Anonymize
        env:
          ANONYMIZATION_SALT: ${{ secrets.ANONYMIZATION_SALT }}
        working-directory: backend
        run: python -m scripts.anonymize_prod_to_staging --source /tmp/prod.sql --output /tmp/staging.sql
      - name: Restore STAGING
        run: psql -f /tmp/staging.sql ...
```

Ajouter un test `test_anonymize_refresh_workflow_has_monthly_schedule` dans `test_github_workflows.py` qui vérifie `wf[on_key]["schedule"][0]["cron"] == "0 2 1 * *"`.

#### MEDIUM-10.7-1 — `aws_s3_bucket_replication_configuration.main` ne dépend pas de l'attachement de la policy IAM

**Fichier** : `infra/terraform/modules/s3/main.tf:200-234`
**Problème** : le `depends_on` actuel ne liste que les versionings source + destination, mais **pas** `aws_iam_role_policy_attachment.replication[0]`. Terraform peut créer la replication configuration en parallèle de l'attachement, et S3 refusera avec `AccessDenied: replication role does not have required permissions` si l'attachement est créé après. La référence `role = aws_iam_role.replication[0].arn` crée une dépendance implicite sur le rôle mais **pas sur la policy attachée**.

**Remédiation** :
```hcl
resource "aws_s3_bucket_replication_configuration" "main" {
  count = var.crr_destination_bucket != null ? 1 : 0
  depends_on = [
    aws_s3_bucket_versioning.main,
    aws_s3_bucket_versioning.destination,
    aws_iam_role_policy_attachment.replication,  # ← AJOUTER
  ]
  ...
}
```

#### MEDIUM-10.7-2 — Replication role exposé au « confused deputy »

**Fichier** : `infra/terraform/modules/s3/main.tf:132-151`
**Problème** : `aws_iam_role.replication.assume_role_policy` accepte `Principal.Service = "s3.amazonaws.com"` **sans** `Condition.StringEquals.aws:SourceAccount` ni `ArnLike.aws:SourceArn = <bucket_arn>`. Pattern confused deputy AWS — le service principal S3 peut être assumé depuis n'importe quel bucket d'un autre compte AWS (peu probable mais durcissement standard recommandé par AWS).

**Remédiation** :
```hcl
resource "aws_iam_role" "replication" {
  count = var.crr_destination_bucket != null ? 1 : 0
  name  = "mefali-${var.env_name}-s3-replication"
  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Service = "s3.amazonaws.com" }
      Action    = "sts:AssumeRole"
      Condition = {
        StringEquals = { "aws:SourceAccount" = data.aws_caller_identity.current.account_id }
        ArnLike      = { "aws:SourceArn" = aws_s3_bucket.main.arn }
      }
    }]
  })
  tags = var.tags
}
```

Ref : https://docs.aws.amazon.com/IAM/latest/UserGuide/confused-deputy.html

#### MEDIUM-10.7-3 — Anti-wildcard CI guard : regex insuffisante

**Fichiers** : `.github/workflows/deploy-dev.yml:58`, `deploy-staging.yml:56`, `deploy-prod.yml:79`
**Problème** : la regex `Resource.*"\*"` ne matche pas :
- `Resource = ["*"]` (tableau),
- `Resource="*"` (sans espace),
- `"arn:aws:s3:::something-*"` (wildcard partiel, type « account root wildcard »),
- multi-line HCL avec jsonencode imbriqué.

La deuxième regex `"arn:aws:s3:::\*"` n'attrape que l'ARN wildcard exact, pas `arn:aws:s3:::xyz*` ni `arn:aws:s3:::*-prod`.

**Impact** : un futur PR introduit un wildcard subtil (ex. `"arn:aws:*"` pour CloudWatch) non détecté.

**Remédiation** : étendre le guard en déplaçant la logique dans un script Python dédié `infra/terraform/scripts/check_no_wildcards.py` qui parse HCL via `python-hcl2` (déjà dep) et valide statement-by-statement. Conserver la regex comme backup mais ajouter le parser.

#### MEDIUM-10.7-4 — `test_no_wildcard_resource_in_iam_module` : assertion fragile

**Fichier** : `backend/tests/test_infra/test_iam_policies.py:87-107`
**Problème** : l'assertion
```python
assert pattern not in text or ":mefali-" in text.split(pattern)[0][-100:]
```
est logiquement équivalente à « soit absent, soit précédé d'un `:mefali-` dans les 100 chars précédents ». Elle est **faux positif-prone** :
- Si quelqu'un introduit `"arn:aws:s3:::autre-bucket/*"` : le pattern `"arn:aws:s3:::*/*"` ne matche pas ce texte, mais `"*"` matche et passe si `:mefali-` existe ailleurs dans les 100 chars (ex. un commentaire).
- Le test passerait même avec un vrai wildcard si un commentaire `# mefali-prod` précède.

**Remédiation** : remplacer par un parser HCL qui isole chaque `Statement.Resource` et assert chacun commence par `arn:aws:s3:::mefali-` ou est un `local.bucket_*`. Les 4 tests actuels reposent aussi sur `text.find("...")` manuel, pattern à refactor en un seul test tabulaire par policy.

#### MEDIUM-10.7-5 — Terraform state bucket partagé entre STAGING et PROD

**Fichiers** : `infra/terraform/envs/staging/backend.tf:4-11`, `infra/terraform/envs/prod/backend.tf:4-10`
**Problème** : les deux envs partagent `bucket = "mefali-terraform-state"` et `dynamodb_table = "mefali-terraform-state-lock"` avec uniquement des `key` différentes. Un IAM compromis STAGING (rôle CI avec `s3:GetObject` sur le bucket state) peut **lire le state PROD** (ARNs, outputs, éventuels password ARNs dans tfvars) via la même permission. L'isolation NFR73 est violée au niveau du state Terraform.

**Remédiation** (choix MVP acceptable, documenter la limite) :
- **Option A (strict)** : 2 buckets séparés `mefali-terraform-state-staging` + `mefali-terraform-state-prod` avec IAM per-env sur chaque bucket.
- **Option B (moyen)** : garder 1 bucket mais scoper IAM via `s3:prefix` dans la policy du rôle CI (lecture `envs/staging/*` seulement pour le rôle staging).

Au minimum : documenter la limite dans `deferred-work.md` + runbook §4 MFA (section existante).

#### LOW-10.7-5 — `_require_salt` utilise `raise SystemExit(message)` — exit code imprédictible

**Fichier** : `backend/scripts/anonymize_prod_to_staging.py:50-57`
