### [CRITICAL-10.1-1] CI ne peut pas exécuter les tests : `psycopg2` manquant dans requirements

**Fichiers** : `backend/requirements-dev.txt`, `.github/workflows/test-migrations-roundtrip.yml:64,75`, `backend/scripts/validate_schema.py:42`, `backend/tests/test_migrations/conftest.py:47-56`

Le workflow CI définit `TEST_ALEMBIC_URL=postgresql+psycopg2://...` et `validate_schema.py` normalise également vers `postgresql+psycopg2://`. Or **ni `psycopg2-binary`, ni `psycopg[binary]`** ne sont présents dans `requirements.txt` (uniquement `asyncpg>=0.30.0`) ni dans `requirements-dev.txt`.

Conséquence : l'étape CI `pytest -m postgres tests/test_migrations/ ...` échouera au premier `create_engine("postgresql+psycopg2://...")` avec `ModuleNotFoundError: No module named 'psycopg2'`. AC10 (gate de merge CI) n'est donc **pas effectif**.

Le Dev Agent Record mentionne une validation locale Docker, mais le workflow GitHub Actions lui-même n'a pu tourner vert que si l'image postgres embarque psycopg2 (elle ne le fait pas — `postgres:16` n'a rien à voir avec le client Python).

**Impact** : bloque merge, invalide la baseline "1283 tests verts" pour le CI réel.

**Patch** — ajouter à `backend/requirements-dev.txt` :

```diff
 # Tests

 pytest-cov>=6.0.0
 aiosqlite>=0.20.0

+psycopg2-binary>=2.9.9  # driver sync requis par tests migrations + validate_schema
```

### [HIGH-10.1-2] `facts.criterion_id` n'a pas de ForeignKeyConstraint vers `criteria.id`

**Fichier** : `backend/alembic/versions/022_create_esg_3_layers.py:45`

La table `facts` déclare la colonne `criterion_id UUID` mais sans `sa.ForeignKeyConstraint(["criterion_id"], ["criteria.id"], ...)`. Toutes les autres tables référencées (`criterion_derivation_rules`:160, `criterion_referential_map`:248, `pack_criteria`:304, `referential_verdicts`:338) ont bien la FK.

**Conséquence** : violation de l'AC7 — « `facts + fact_versions` : **FK** + index sur `(company_id, criterion_id)` ». Permet d'insérer un `fact` pointant vers un criterion inexistant → cohérence D3 (ESG 3 couches) compromise.

**Patch** :

```diff
         sa.ForeignKeyConstraint(
             ["company_id"], ["companies.id"], ondelete="CASCADE"
         ),
+        sa.ForeignKeyConstraint(
+            ["criterion_id"], ["criteria.id"], ondelete="SET NULL"
+        ),
         sa.ForeignKeyConstraint(
             ["source_document_id"],
             ["documents.id"],
             ondelete="SET NULL",
         ),
```

`ondelete="SET NULL"` plutôt que `CASCADE` : `criterion_id` est déjà `nullable=True`, et la suppression d'un critère ne doit pas supprimer les faits historiques (audit trail FR19).

### [HIGH-10.1-3] Condition dupliquée dans `_to_sync_url` — typo psycopg/psycopg2

**Fichier** : `backend/tests/test_migrations/conftest.py:49-51`

```python
if url.startswith("postgresql+psycopg://") or url.startswith(
    "postgresql+psycopg://"
):
    return url
```

Les deux opérandes du `or` sont **identiques**. Le typo évident est `"postgresql+psycopg2://"` pour la seconde branche. Avec `TEST_ALEMBIC_URL=postgresql+psycopg2://...` (CI), l'URL tombe à travers toutes les branches et est renvoyée telle quelle — ça fonctionne actuellement par accident, mais :
- Le docstring ligne 47 dit `(psycopg2)` → intent d'origine.
- Si un jour un contributeur ajoute une normalisation dans le `return url` final, les URLs `psycopg2://` repartiraient non normalisées.

**Patch** :

```diff
-    if url.startswith("postgresql+psycopg://") or url.startswith(
-        "postgresql+psycopg://"
-    ):
+    if url.startswith("postgresql+psycopg2://") or url.startswith(
+        "postgresql+psycopg://"
+    ):
         return url
```

### [HIGH-10.1-4] Inconsistance driver sync : `conftest` → `psycopg`, `validate_schema` → `psycopg2`

**Fichiers** : `backend/tests/test_migrations/conftest.py:54,56` vs `backend/scripts/validate_schema.py:42`

`conftest.py` normalise `asyncpg://` et `postgresql://` → `postgresql+psycopg://` (driver v3).
`validate_schema.py` normalise vers `postgresql+psycopg2://` (driver v2).

Sur le même `TEST_ALEMBIC_URL`, les deux modules utilisent des drivers différents. Si `requirements-dev.txt` n'installe que l'un des deux, l'autre cassera. Cohérence recommandée : choisir **psycopg2-binary** (retenu par CI + validate_schema, plus ancien, compatibilité large) et aligner conftest.

**Patch** (sur `conftest.py:54,56`) :

```diff
     if url.startswith("postgresql+asyncpg://"):
-        return "postgresql+psycopg://" + url[len("postgresql+asyncpg://") :]
+        return "postgresql+psycopg2://" + url[len("postgresql+asyncpg://") :]
     if url.startswith("postgresql://"):
-        return "postgresql+psycopg://" + url[len("postgresql://") :]
+        return "postgresql+psycopg2://" + url[len("postgresql://") :]
```

Et `conftest.py:49` : `or url.startswith("postgresql+psycopg2://")` dans la condition de pass-through (fix combiné avec HIGH-10.1-3).

### [MEDIUM-10.1-5] Backfill `fund_applications.project_id` : ambiguïté multi-FA par user

**Fichier** : `backend/alembic/versions/020_create_projects_schema.py:310-324`

La clause `UPDATE fund_applications fa SET project_id = (... LIMIT 1)` sélectionne **un** project par user distinct. Si un user a 3 fund_applications legacy et 2 projects pré-existants (cas hors-MVP mais théoriquement possible si une story antérieure a créé des projects), toutes ses FA sont rattachées à **le même** project aléatoire. Comportement surprenant, non documenté dans la docstring.

**Mitigation** : ajouter `ORDER BY p.created_at ASC` pour déterminisme et une note docstring. Alternative : créer un project par FA (explose les lignes mais préserve l'unicité d'intention).

**Patch minimal** :

```diff
             sa.text(
                 """
                 UPDATE fund_applications fa
                 SET project_id = (
                     SELECT p.id
                     FROM projects p
                     JOIN companies c ON p.company_id = c.id
                     WHERE c.owner_user_id = fa.user_id
+                    ORDER BY p.created_at ASC
                     LIMIT 1
                 )
                 WHERE fa.project_id IS NULL
                 """
             )
```

Et compléter le docstring du fichier : « Backfill déterministe : chaque FA est rattachée au project le plus ancien du même user. »

## Layer 2 — Security & Data Safety

### [HIGH-10.1-11] `admin_access_audit` et `admin_catalogue_audit_trail` — pas de protection tamper

**Fichiers** : `024_enable_rls_on_sensitive_tables.py:66-117`, `026_create_admin_catalogue_audit_trail.py:34-99`

Les tables d'audit D6/D7 sont supposées « immuables (write-only + read admin_super) » (Dev Notes ligne 222). Mais côté migration :
- Aucun `REVOKE UPDATE, DELETE` sur ces tables.
- Pas de policy RLS empêchant UPDATE/DELETE.
- Pas de trigger `BEFORE UPDATE/DELETE → RAISE EXCEPTION`.

Un `admin_super` (ou tout rôle avec DELETE) peut supprimer des lignes d'audit → brèche de conformité FR64 (rétention 5 ans).

Le Dev Agent Record et `deferred-work.md` mentionnent que l'**event listener** qui écrit est livré Story 10.5, mais rien sur la **protection écriture**. Recommandation : ajouter à `deferred-work.md` :

> **Audit tables tamper-proofing** (D6/D7) — livré **Story 10.5** : `REVOKE UPDATE, DELETE ON admin_access_audit, admin_catalogue_audit_trail FROM PUBLIC;` + trigger `RAISE EXCEPTION 'audit immuable'` sur UPDATE/DELETE pour tout rôle hors `admin_super`. Infra table créée en 10.1.

Non bloquant mais à tracer explicitement.

### [MEDIUM-10.1-12] RLS policies — pas de `WITH CHECK` explicite

**Fichier** : `024_enable_rls_on_sensitive_tables.py:128-132`

```sql
CREATE POLICY tenant_isolation ON <table>
  FOR ALL USING (<predicate>)
```

Postgres ≥ 10 applique automatiquement l'expression `USING` comme `WITH CHECK` si ce dernier est absent (cf. [docs Postgres 16 CREATE POLICY](https://www.postgresql.org/docs/16/sql-createpolicy.html)). Le comportement actuel est donc sécurisé, mais **implicite** — un reviewer futur peut mal interpréter.

**Patch recommandé** pour explicitation :

```python
op.execute(
    sa.text(
        f"CREATE POLICY tenant_isolation ON {table} "
        f"FOR ALL USING ({using}) WITH CHECK ({using})"
    )
)
```

### [MEDIUM-10.1-13] Backfill 020 — risque de fund_applications orphelins si user supprimé

**Fichier** : `backend/alembic/versions/020_create_projects_schema.py:286-324`

Si un `fund_applications.user_id` pointe vers un `users.id` **déjà supprimé** (référence orpheline préexistante), la clause `WHERE c.owner_user_id = fa.user_id` ne trouve aucune company (CASCADE aurait déjà supprimé les FA, sauf s'il n'y a pas de FK ON DELETE CASCADE). Dans ce cas, `fa.project_id` reste NULL → `ALTER COLUMN SET NOT NULL` **échoue**, bloquant le upgrade.

Mitigation : vérifier avant le SET NOT NULL qu'il n'y a pas de NULL résiduels, et si oui échouer explicitement (message clair) ou créer un bucket « orphan » :

```python
# Garde-fou avant SET NOT NULL
if dialect == "postgresql":
    result = op.get_bind().execute(
        sa.text("SELECT COUNT(*) FROM fund_applications WHERE project_id IS NULL")
    ).scalar()
    if result:
        raise RuntimeError(
            f"{result} fund_applications sans project_id après backfill — "
            "user_id orphelin probable, nettoyer les données legacy avant upgrade."
        )
```

Laissé à la discrétion du PM (en l'état MVP sans données prod, le risque est théorique).

### Q-A : `test_hallucinated_summary_triggers_retry_then_fails` lié 10.1 ?

### Q-B : Backfill company+project legacy par user distinct — robuste ?

**Partiellement**. Les edge cases gérés :
- ✅ User sans FA → pas de company créée (OK).
- ✅ User avec FA mais sans company existante → legacy company + project créés.
- ✅ User avec company existante → nouveau project legacy rattaché (WHERE NOT EXISTS).

Edge cases non gérés :
- ⚠️ User avec plusieurs FA → toutes rattachées au même project (cf. MEDIUM-10.1-5).
- ⚠️ User orphelin (référence FK cassée) → blocage SET NOT NULL (cf. MEDIUM-10.1-13).

**Non définie** dans 10.1. Voir MEDIUM-10.1-6 pour tracer explicitement vers Story 10.10 ou 20.4.
