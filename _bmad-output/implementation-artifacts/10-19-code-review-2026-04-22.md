### HIGH (4)

#### H1 — `displayValue` manquant sur `ComboboxInput` en mode single (UX runtime bug)
- **Source** : edge (renforce blind B8)
- **Location** : `frontend/app/components/ui/Combobox.vue:214-215`
- **Evidence** :
  ```vue
  <ComboboxInput
    v-model="searchTerm"
    @compositionstart="handleCompositionStart"
    @compositionend="handleCompositionEnd"
    ...
  />
  ```
- **Mécanisme** : Reka `ComboboxInput.js:81` sans `displayValue` prop fait `modelValue.value = rootModelValue.toString()` à la sélection. Après choix de "Sénégal" (value=`'sn'`), l'input affiche la **clé brute `'sn'`** au lieu du label "Sénégal", et `searchTerm` devient `'sn'` — à la réouverture, `filteredOptions` filtre avec `'sn'` et masque la plupart des options.
- **Patch** : ajouter `:display-value="(v) => labelFor(v as T)"` sur `ComboboxInput`.
- **Test manquant** : round-trip select → close → open affiche label + options non filtrées.

#### H2 — `searchTerm` non réinitialisé au close-without-select
- **Source** : edge
- **Location** : `frontend/app/components/ui/Combobox.vue:101-186` (pas de watcher `open`)
- **Evidence** :
  ```ts
  const searchTerm = ref('');
  // Aucun watch(() => props.open) qui reset au close
  ```
- **Mécanisme** : user tape `'zzz'` → pas de résultat → Escape/click-outside → réouverture montre directement empty state (stale `'zzz'`). Chaque consommateur qui n'ajoute pas un reset externe subit ce bug.
- **Patch** :
  ```ts
  watch(() => props.open, (newOpen) => {
    if (!newOpen) searchTerm.value = '';
  });
  ```
  Ou dans `@update:open` handler avec équivalent explicite.

#### H4 — Tests AC4 `aria-controls` + `aria-activedescendant` absents
- **Source** : auditor
- **Location** : `frontend/tests/components/ui/test_combobox_a11y.test.ts` (lignes 3054-3114)
- **Evidence** : AC4 spec ligne 439 exige explicitement `.getAttribute('aria-controls')` non-null + `aria-activedescendant` après ArrowDown. Aucune assertion `toHaveAttribute('aria-controls', ...)` ni `toHaveAttribute('aria-activedescendant', ...)` dans le fichier livré.
- **Mécanisme** : deux attributs ARIA nommés dans AC4 sans coverage auto. Leçon 21 §4quater (tests observables ≠ smoke) enfreinte — l'auteur n'a pas listé cet écart dans les Completion Notes, donc Leçon 20 §4quater (écarts vs spec obligatoires) également enfreinte.
- **Patch** : ajouter 2 tests DOM observables :
  ```ts
  it('AC4 : aria-controls référence le listbox ouvert', async () => {
    ...
    expect(combobox).toHaveAttribute('aria-controls', expect.stringMatching(/reka-combobox-content-/));
  });
  it('AC4 : aria-activedescendant suit ArrowDown', async () => {
    await user.click(trigger);
    await user.keyboard('{ArrowDown}');
    expect(combobox.getAttribute('aria-activedescendant')).not.toBeNull();
  });
  ```

#### H5 — AC14 coverage ≥ 85% non mesurée instrumentée
- **Source** : auditor
- **Location** : `_bmad-output/.../10-19-....md:1244` Completion Notes
- **Evidence** :
  > Coverage : non mesurée séparément (pas d'outil coverage configuré Phase 0). Densité tests observée = 90 tests… couverture effective ≥ 85% confirmée par audit ligne-par-ligne
- **Mécanisme** : AC14 exige une valeur mesurée, pas réclamée. « Audit ligne-par-ligne » n'est pas un substitut instrumenté.
- **Patch** (options) :
  1. Exécuter `vitest --coverage` sur `Combobox.vue` + `Tabs.vue` seuls et reporter la valeur réelle dans Completion Notes.
  2. OU documenter explicitement la substitution comme **écart vs spec** (Leçon 20) avec un DEF ticket pour instrumenter Phase 1.

### MEDIUM (6)

#### M1 — Test Badge count couplé à `aria-label="Statut Brouillon"` (Leçon 21 fragile)
- **Source** : blind+edge+auditor (MERGED)
- **Location** : `test_combobox_behavior.test.ts:109` (≈ ligne 3229 du diff)
- **Evidence** :
  ```ts
  const badges = screen.getAllByRole('img', { name: /Statut Brouillon/i });
  expect(badges.length).toBe(2);
  ```
- **Impact** : le test fait un count mais ne vérifie pas que les labels visibles ("Sénégal", "Côte d'Ivoire") apparaissent **dans** chaque badge. Si `labelFor()` fallback sur `String(val)`, le test passe quand même. Couplage à un `aria-label` latéral du composant `Badge` (sensible au shim DEF-10.19-4).
- **Patch** : ajouter `expect(screen.getByText('Sénégal')).toBeInTheDocument()` + `expect(screen.getByText(/côte d'ivoire/i)).toBeInTheDocument()`.

#### M2 — Tests IME utilisent `input.value = 'sen'` impératif (anti-Pattern A)
- **Source** : blind+edge (MERGED)
- **Location** : `test_combobox_behavior.test.ts:218-224`
- **Evidence** :
  ```ts
  input.value = 'sen'; // direct DOM write — pas d'event 'input'
  await fireEvent.compositionEnd(input, { target: input });
  ```
- **Impact** : violation Pattern A (DOM observable pas imperative). La réactivité Vue ne voit pas ce write ; le test ne passe que parce que `ignoreFilter=true` masque la désync Reka-side. Aucune garantie que le chemin composition→filter marche en prod.
- **Patch** : remplacer par `await user.type(input, 'sen')` **pendant** la composition, ou utiliser `fireEvent.input(input, { target: { value: 'sen' } })` puis `fireEvent.compositionEnd(...)`.

#### M3 — AC3 IME test utilise `.not.toBeNull()` proxy au lieu du `.toBeNull()` strict prescrit
- **Source** : auditor
- **Location** : `test_combobox_behavior.test.ts:3311-3325`
- **Evidence** : spec AC3 Task 5 ligne 439 prescrit :
  > `expect(screen.queryByRole('option', {name: /sénégal/i})).toBeNull()` strict
  L'impl utilise `.not.toBeNull()` sur `burkina faso` (qui n'a ni `e` ni `é`, donc passe même sans guard IME).
- **Impact** : le test passe **même si `isComposing` ne monte jamais à true**. Observabilité Leçon 21 cassée.
- **Patch** : revenir à l'assertion prescrite `.toBeNull()` sur `sénégal` pendant composition.

#### M4 — Badge sm (`max-h-[24px]`) + bouton × `min-h-11 min-w-11` (44 px) → overflow visuel
- **Source** : edge
- **Location** : `Combobox.vue:191-212`
- **Evidence** :
  ```vue
  <Badge variant="lifecycle" state="draft" size="sm">
    <span class="inline-flex items-center gap-1">
      ...
      <button class="... min-h-11 min-w-11 ...">
  ```
- **Impact** : le bouton 44 px force la hauteur flex-row, le `max-h-[24px]` du Badge sans `overflow:hidden` est dépassé. Chaque badge sélectionné = 44 px de haut — l'ancre combobox devient monstrueuse dès 2+ sélections.
- **Patch** (options) :
  1. Réduire le bouton à 20-24 px (`min-h-5 min-w-5`) et laisser la zone tactile déborder via pseudo-élément `::before` étendu (WCAG 2.5.5).
  2. OU utiliser un Badge `md` dédié multi-select avec touch-target conforme.

#### M5 — `aria-label` identique "Statut Brouillon" sur tous badges multi-select (a11y)
- **Source** : edge
- **Location** : `Combobox.vue:194-197` + `Badge.vue:46`
- **Evidence** : chaque Badge sélectionné hard-code `variant="lifecycle" state="draft"` → `Badge.vue` calcule `aria-label="Statut Brouillon"` pour tous.
- **Impact** : screen reader entend "Statut Brouillon" × N sans distinguer les options. Le × button porte `aria-label="Retirer Sénégal"` correct, mais le chip lui-même est opaque. WCAG 1.1.1 (Name, Role, Value) borderline.
- **Patch** : surcharger `aria-label` du Badge avec le label dynamique, ou retirer le `state="draft"` en passant un Badge neutre (`variant="neutral"`), ou utiliser un span simple avec `aria-label={label}`.

#### M6 — `isComposing` reste `true` si Escape annule la composition IME sur Chrome macOS
- **Source** : edge
- **Location** : `Combobox.vue:147-157`
- **Evidence** : Chrome macOS annule une composition sur Escape **sans émettre `compositionend`** (inconsistance cross-browser documentée WHATWG).
- **Impact** : `filteredOptions` retourne `props.options` inconditionnellement → search cassé silencieusement après un IME cancel. Niche mais réel pour utilisateurs CJK.
- **Patch** : ajouter `@keydown.escape` handler sur `ComboboxInput` qui reset `isComposing.value = false`, et/ou watcher sur `open` qui flush `isComposing` au close.

## Leçons §4quinquies candidates (7ème pattern émergent)

Sur cette 20ème review Phase 4 (2ème et 3ème wrapper Reka UI simultanés), deux patterns récurrents se stabilisent et méritent d'être remontés en §4quinquies :

### Leçon 22 §4quinquies — Wrapper Reka UI : `displayValue` obligatoire sur `ComboboxInput` single-select

**Constat** : 2 stories sur 2 wrappers Combobox-like (10.18 Drawer header + 10.19 Combobox) ont omis une prop Reka UI qui gouverne l'affichage post-sélection. Reka par défaut fait `value.toString()` ce qui casse l'UX quand `value ≠ label`.

**Règle** : tout wrapper Reka UI qui expose un contrôleur de saisie **doit** passer un callback `displayValue` si le `value` n'est pas le label rendu à l'utilisateur. Documenter dans `docs/CODEMAPS/ui-primitives.md` comme piège **#41**.

**Acceptance test** : round-trip `select → close → reopen` observe le label (pas la clé) dans l'input ET la liste complète des options (pas filtrée par la clé).

### Leçon 23 §4quinquies — Wrapper Reka UI : `searchTerm` lifecycle au close

**Constat** : les wrappers qui gèrent un état local de recherche (`searchTerm` ref) doivent explicitement watch l'état `open` pour reset au close-without-select. Reka `resetSearchTermOnSelect` ne couvre pas le cas "close sans select".

**Règle** : tout wrapper Reka UI avec state de search local → watcher `@update:open` (false) → reset du state.

**Acceptance test** : ouvrir, taper `zzz`, fermer (Escape ou click-outside), rouvrir → liste complète visible, input vide.

### Leçon 24 §4quinquies — `aria-controls` + `aria-activedescendant` : tests strict-attribute, pas proxy

**Constat** : sur 3 stories a11y (10.15, 10.17, 10.19), les tests ARIA tendent à utiliser des queries `getByRole('option', { name: ... })` en proxy au lieu de `.toHaveAttribute('aria-controls', ...)` strict. Le test passe quand l'attribut est absent si un autre mécanisme (focus, DOM) happens to donner le même résultat.

**Règle** : quand une AC cite nominativement un attribut ARIA → le test doit le matcher **directement** via `toHaveAttribute` (ou `.getAttribute(...) !== null`), pas via effet de bord.
