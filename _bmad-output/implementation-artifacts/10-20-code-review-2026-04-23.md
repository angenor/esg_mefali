## 3. Findings HIGH (3) — À corriger avant merge

### H-1 — Conflit `aria-controls` : `popoverId` Vue écrasé par `contentId` Reka — code mort + test confirme l'override

- **Source :** Blind F3 + Edge E1 + Edge E9 (triple corroboration)
- **Fichier :** `frontend/app/components/ui/DatePicker.vue:284,287` + `test_datepicker_a11y.test.ts:99`
- **Leçon violée :** L24 §4quinquies (ARIA attribute-strict — valeur littérale)
- **Évidence :**
  ```vue
  <PopoverTrigger :aria-controls="popoverId" ...>
  <PopoverContent :id="popoverId" role="dialog" aria-modal="false" ...>
  ```
  Le test strict est `ariaControls.toMatch(/reka-popover-content-/)` — ce qui confirme que la valeur sur le DOM rendu provient de Reka UI `rootContext.contentId`, **pas** du `popoverId` Vue `useId()` passé explicitement. En d'autres termes : `aria-controls` sur le trigger pointe vers l'id Reka (OK fonctionnellement), mais la prop `:aria-controls="popoverId"` dans le template est du **code mort**. Symétrique côté content : `:id="popoverId"` peut être écrasé par l'id interne Reka.
- **Risque :** si une upgrade Reka UI change l'ordre de binding des forwarded props, le lien `aria-controls` peut pointer vers un id qui n'existe pas → AT ne résolvent pas le dialogue.
- **Correctif :** retirer `const popoverId = useId()` + `:aria-controls="popoverId"` + `:id="popoverId"`. Laisser Reka UI gérer le wiring natif. Adapter le test a11y pour affirmer strictement `ariaControls === content.id` (cohérence DOM) plutôt qu'un regex `/reka-popover-content-/`.

### H-2 — `currentMonth` non réactif aux mutations externes de `modelValue` (watch manquant)

- **Source :** Blind F2 + Edge E4
- **Fichier :** `frontend/app/components/ui/DatePicker.vue:2490-2512`
- **AC violé :** AC7 (lifecycle close) extension implicite
- **Évidence :** `watch(isOpen, ...)` restaure `currentMonth` uniquement au close sans sélection. Il n'existe **aucun** `watch(() => props.modelValue, ...)` pour synchroniser `currentMonth` quand le parent met à jour `modelValue` via v-model (ex. reset formulaire, correction programmatique).
- **Risque :** consommateur efface la valeur via l'API (reset) pendant que le popover est ouvert, ou change la valeur avant réouverture — le DatePicker affiche le mois de l'ancienne valeur. Divergence visuelle/sémantique entre valeur interne et valeur externe.
- **Correctif :**
  ```ts
  watch(
    () => props.modelValue,
    () => { if (!isOpen.value) currentMonth.value = initialMonth() },
    { deep: true },
  );
  ```
  Ajouter un test Vitest observable : `wrapper.setProps({ modelValue: null }); wrapper.find('button').trigger('click'); expect(getHeading().textContent).toMatch(/aujourd'hui/)`.

### H-3 — Test AC5 `PageDown` faux positif : `dispatchEvent` impératif + assertion `/(avril|mai) 2026/` toujours vraie

- **Source :** Blind F5 + Edge E2 + Auditor A4 (triple corroboration)
- **Fichier :** `frontend/tests/components/ui/test_datepicker_behavior.test.ts:3548-3570`
- **Leçon violée :** L21 §4quater (assertions observables strictes, Pattern A user-event)
- **Évidence :**
  ```ts
  const event = new KeyboardEvent('keydown', { key: 'PageDown', bubbles: true });
  cellFocused.dispatchEvent(event);
  app.dispatchEvent(event);  // mutation stateful, même objet dispatché 2×
  await new Promise((r) => setTimeout(r, 50));
  expect(getHeading().textContent?.toLowerCase()).toMatch(/(avril|mai) 2026/);
  ```
  L'assertion `/(avril|mai)/` est satisfaite si PageDown ne fait **rien** (heading reste sur `avril 2026`). Le test prouve uniquement que le heading affiche un mois valide — il ne prouve **pas** que PageDown fonctionne.
- **Risque :** régression silencieuse garantie. Si Reka UI 3.x retire la gestion PageDown ou change le focus bubbling, ce test passe vert.
- **Correctif :** deux options équivalentes :
  1. **Délégation formelle Storybook** : remplacer le test par un commentaire `// DELEGATED TO Storybook story "DatePicker/SingleKeyboardPageDown" — happy-dom focus bubbling limitation` + créer la story Storybook correspondante avec test runner.
  2. **Remplacer par un fallback observable CalendarNext click** :
     ```ts
     await user.click(document.querySelector('[aria-label*="mois suivant"]'));
     await nextTick();
     expect(getHeading().textContent?.toLowerCase()).toBe('mai 2026');  // strict
     ```
  L'assertion stricte `toBe('mai 2026')` (pas `toMatch(/avril|mai/)`) est impérative pour catch la régression.

### M-1 — AC5 Keyboard : 9/12 comportements absents de Vitest et non documentés comme délégués per-path

### M-2 — AC8 : `maxValue` (CalendarNext disabled) et `readonly` (Enter no-op) absents des tests Vitest

- **Source :** Auditor A2
- **AC violé :** AC8 (5 comportements documentés → 3 testés)
- **Évidence :** suite AC8 (lignes 3674-3815) couvre `minValue` ✅, `isDateDisabled weekends` ✅, `disabled-prevents-open` ✅. Mais `maxValue CalendarNext disabled when month > maxValue` et `readonly: true → popover ouvre + Enter no-op` ne sont pas testés. La Completion Notes ne documente pas ces omissions dans « Ajustements mineurs vs spec ».
- **Correctif :** ajouter 2 tests observables. Le code du composant passe bien `maxValue` et `readonly` à `CalendarRoot` — il suffit d'ajouter la couverture.

### M-3 — `displayValue` range : état `{ start: null, end: DateValue }` produit un affichage inversé `'Fin ? — 30/04/2026'`

- **Source :** Blind F4 + Edge E5
- **Fichier :** `frontend/app/components/ui/DatePicker.vue:2525-2534`
- **Évidence :** la gate `if (!start && !end) return null` ne protège pas contre `start=null + end=DateValue`. `startStr` vaut `props.rangePartialLabel` (`'Fin ?'`), `endStr` vaut une date formatée → trigger affiche `'Fin ? — 30/04/2026'` (ordre sémantique inversé). Aucun test ne couvre cet état.
- **Correctif :** normaliser en amont ou ajouter une gate explicite :
  ```ts
  if (!start && end) return `${props.rangePartialLabel} — ${formatter.value.format(end.toDate(...))}`;
  if (start && !end) return `${formatter.value.format(start.toDate(...))} — ${props.rangePartialLabel}`;
  ```
  Ou strictement : `if (!start) return null` (refuser l'état incohérent). Ajouter test range partiel.

### M-4 — `handleSingleChange` : mutation directe de `currentMonth` + fermeture sur re-sélection de la même date

- **Source :** Edge E3
- **Fichier :** `frontend/app/components/ui/DatePicker.vue:2503-2512`
- **Violation :** pattern coding-style.md (immutabilité) + UX subtle
- **Évidence :**
  ```ts
  function handleSingleChange(newValue: DateValue | null | undefined) {
    const val = newValue ?? null;
    emit('update:modelValue', val);
    if (val) {
      currentMonth.value = val;  // mutation directe
      isOpen.value = false;
    }
  }
  ```
  Si l'utilisateur clique sur la date déjà sélectionnée (même valeur), Reka UI ré-émet `update:model-value`. Le composant re-émet + ferme → comportement possiblement intentionnel mais non documenté. Pattern mutation directe plutôt que via computed/immutable.
- **Correctif :** soit documenter le comportement dans dev-notes (double-clic same date = confirm+close), soit gate : `if (val && !isSameDate(val, props.modelValue)) { currentMonth.value = val }`.

### M-5 — Test `aria-disabled` sur `gridcell` parent plutôt que sur `CalendarCellTrigger`

- **Source :** Blind F8
- **Fichier :** `frontend/tests/components/ui/test_datepicker_behavior.test.ts:3774-3778`
- **AC violé :** AC4 (prescription `CalendarCellTrigger` porte `aria-disabled="true"`)
- **Évidence :** `const parent = cell5.closest('[role="gridcell"]'); expect(parent?.getAttribute('aria-disabled')).toBe('true')`. L24 §4quinquies exige des assertions strict-attribute sur le bon élément. Si Reka UI déplace `aria-disabled` vers le button dans une future version, le test passera sur le mauvais élément.
- **Correctif :** tester **les deux** :
  ```ts
  const trigger = cell5.querySelector('button, [role="button"]');
  expect(trigger?.getAttribute('aria-disabled')).toBe('true');  // strict sur trigger
  ```

## 9. Leçons §4sexies candidates (25ᵉ+ pattern émergent)

Cette review révèle 3 patterns méritant capitalisation :

### Leçon 25 candidate — **Reka UI : ne pas injecter d'`id` custom sur des primitives slot-forwarded**

- **Contexte :** H-1 a démontré que `PopoverContent :id="popoverId"` est écrasé par `rootContext.contentId` Reka, rendant le binding custom du `aria-controls` trigger du code mort.
- **Règle :** pour les primitives Reka UI qui forward un id interne via contexte (Popover, Combobox, Dropdown), **laisser Reka gérer le wiring natif** — ne pas injecter de `useId()` Vue.
- **Test d'observabilité :** `expect(trigger.getAttribute('aria-controls')).toBe(content.id)` (cohérence DOM, pas regex).
- **Application :** stories 10.21+ wrappers Reka UI (Menu, DropdownMenu, ContextMenu).

### Leçon 26 candidate — **Délégation Storybook per-path, pas globale**

- **Contexte :** M-1 a identifié que « keyboard AC5 délégué Storybook » est documenté globalement en Completion Notes, mais 9 chemins individuels ne sont pas marqués per-path.
- **Règle :** chaque chemin AC délégué doit avoir un commentaire inline `// DELEGATED TO Storybook <story-name> — <raison technique>` dans le fichier test, ET une story Storybook correspondante.
- **Test d'observabilité Vitest :** `describe.skip('AC<n> <path> — DELEGATED Storybook <story-id>', () => { it.todo('covered in Storybook') })`.
- **Application :** toute limitation happy-dom/jsdom future (focus bubbling, IntersectionObserver, ResizeObserver, CSS computed).

### Leçon 27 candidate — **Ordonnancement pièges cross-story : continuité séquentielle > alignement spec**

- **Contexte :** I-2 documente écart pièges #42-45 vs spec #41-44 (justifié par collision avec #41 Combobox 10.19).
- **Règle :** lors de review de spec en brownfield, le dev scanne `docs/CODEMAPS/*.md` pour détecter le plus haut numéro piège existant **avant** d'ouvrir la story, puis décale la numérotation proposée. Éviter la renumérotation en fin de dev.
- **Test d'observabilité :** ajouter `expect(uniqueNumbers.size === matches.length).toBe(true)` au lieu de `expect(matches.length >= N)` — test d'unicité + continuité.
- **Application :** toutes stories futures ajoutant des pièges §5 CODEMAPS.
