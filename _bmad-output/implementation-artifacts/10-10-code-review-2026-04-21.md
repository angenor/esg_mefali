## LAYER 2 — CONCURRENCY (focus principal)

### ✅ Points validés

1. **`FOR UPDATE SKIP LOCKED`** — Test `test_worker_skip_locked_allows_concurrent_processing` (`test_worker_e2e.py:88-128`) insère 10 events et lance 2 coroutines `process_outbox_batch` en parallèle. Chaque event traité exactement 1 fois (`status='processed'`, `retry_count=0`). ✓ *Caveat* : asyncio coopératif dans un seul process — voir MEDIUM-10.10-3.
2. **`AsyncIOScheduler max_instances=1 + coalesce=True`** — impossible de superposer 2 exécutions du même job (`worker.py:284-286, 293-295`). Coalesce écarte les ticks ratés (backlog mis à zéro au prochain tick, pas de « run 3× pour rattraper »).
3. **Timezone UTC explicit partout** : scheduler `timezone="UTC"` + migration `TIMESTAMPTZ DEFAULT now()` (PG stocke nativement UTC) + `_utcnow()` timezone-aware + logs `isoformat()` avec offset `+00:00`. Cohérence end-to-end.
4. **Session pool release** — `async with AsyncSession(engine, ...) as db:` systématique dans `process_outbox_batch` et `purge_expired_prefill_drafts`. `__aexit__` libère le slot pool quel que soit le chemin (succès / exception).
5. **Lifespan shutdown ordre** — `main.py:93-96` : `await stop_outbox_scheduler(...)` **avant** que le pool de connexions ne ferme (commentaire explicite). `scheduler.shutdown(wait=True)` attend la fin des jobs en cours. ✓
6. **`next_retry_at` filtre applicatif** — la query `or_(next_retry_at.is_(None), next_retry_at <= func.now())` vit dans le worker (pas dans le DDL d'index, `now()` non-IMMUTABLE). Migration 029 documente explicitement ce choix. ✓
7. **Dead-letter après `MAX_RETRIES=3`** — `status='failed'` + `processed_at=now()` → sort du filtre `processed_at IS NULL` → **jamais repêché**. Test `test_worker_marks_failed_after_3_retries` prouve qu'une 4ᵉ passe ne ré-incrémente pas `retry_count`. ✓
8. **Race retry transient** — worker A crash avant UPDATE : au commit manqué, verrou row-level est relâché (rollback implicite). Worker B pick le même event via SKIP LOCKED. Comportement correct par design PostgreSQL. ✓
9. **Handler registry import-time** — `_validate_unique_event_types()` ligne 140 est safe à rappeler (scan linéaire, pas d'effet de bord). Test `test_validate_unique_event_types_detects_duplicates` patch le tuple et re-valide, sans pollution de l'import initial. ✓
10. **`SystemExit` / `BaseException` propagation** — `dispatch_event` capture uniquement `Exception` (`handlers.py:170`). `BaseException` propage au scheduler → le process meurt proprement. Documenté §5 Pièges #7 et handler contract ligne 17. ✓

### 🔶 Points de vigilance

**HIGH-10.10-1 — Session partagée batch : `PendingRollbackError` si handler SQL raise après écriture partielle**

Le worker ouvre **une seule session partagée** pour tout le batch et la passe aux handlers (`worker.py:106 → dispatch_event → entry.handler(event, db)`).

**Scénario à risque** :
```python
async def epic13_handler(event: DomainEvent, db: AsyncSession) -> None:
    await db.execute(update(ReferentialVerdict).where(...))  # ← succès
    raise RuntimeError("transient error")                    # ← session -> bad state
```

Conséquence :
1. SQLAlchemy marque la transaction courante comme `PendingRollback`.
2. Le worker continue la boucle : `event.retry_count = 1` (in-memory, OK).
3. À la fin du batch : `await db.commit()` → `PendingRollbackError` levée.
4. Le `async with` rollback → **aucune ligne mise à jour** : l'event garde `retry_count=0, next_retry_at=NULL`.
5. Au tick suivant (30 s), même event repêché, même échec → **hot-loop** jusqu'à ce que la condition transitoire se résolve.

**Pourquoi MVP non-bloquant** : le seul handler actuel (`noop.test`) ne touche pas la session. Epic 13/14 introduiront les premiers handlers métier manipulant `facts`, `referential_verdicts`, `mv_fund_matching_cube`.

**Remédiation (Epic 13 pré-requis)** :
- Option A (recommandée) — savepoint par event : `async with db.begin_nested():` autour de `dispatch_event`.
- Option B — session dédiée par event : `async with AsyncSession(engine) as per_event_db:` dans la boucle.
- Option C — limiter les handlers à des opérations sans effet BDD direct (invalidation cache externe, pub/sub) et dériver toute écriture via un writer Outbox secondaire.

[backend/app/core/outbox/worker.py:106-132]

**MEDIUM-10.10-1 — `BACKOFF_SCHEDULE[2]=600s` dead code avec `MAX_RETRIES=3`**

```python
BACKOFF_SCHEDULE = (30, 120, 600)  # §D11 architecture
MAX_RETRIES = 3

# worker.py:186-210
event.retry_count += 1                        # 1, 2, 3
if event.retry_count >= MAX_RETRIES:          # 3 → failed
    event.status = "failed"; return
# Retry : index = retry_count - 1 → 0 (30s), 1 (120s). JAMAIS 2.
```

Le 3ᵉ élément `600 s` n'est atteignable que si `MAX_RETRIES >= 4`. L'architecture.md §D11 affiche `[30, 120, 600]` — le code délivre seulement 2 retries effectifs. Le test `test_worker_marks_failed_after_3_retries` confirme 3 attempts total (2 retries scheduled).

**Remédiation (au choix)** :
- `BACKOFF_SCHEDULE = (30, 120)` + update docs §D11 + outbox.md §3.
- `MAX_RETRIES = 4` + update test `test_worker_marks_failed_after_3_retries` → 4 retries.

Coût minime, mais divergence doc/impl détectable à tout moment par un reviewer externe. Candidat correctif direct 10.11 OU deferred-work Epic 20.

[worker.py:55-59, 186-221]

**MEDIUM-10.10-2 — `purge_expired_prefill_drafts` ne boucle pas sur plusieurs batches**

```python
PREFILL_PURGE_BATCH_SIZE: Final[int] = 500
# worker.py:229-256 : 1 seul DELETE ... LIMIT 500 par tick horaire.
```

Si > 500 drafts expirés (incident, pic trafic, backlog accumulé pendant panne), seulement 500 purgés par heure → résorption lente (1000 drafts expirés → 2 h, 5000 → 10 h). La table gonfle temporairement sans garde-fou métrique.

**Remédiation** : boucle `while rowcount == PREFILL_PURGE_BATCH_SIZE` avec cap sécurité (10 itérations max par tick = 5000 drafts/tick) + log `metric=prefill_drafts_purge_saturated` si cap atteint. 8 lignes de code. Candidat patch LOW opportuniste.

[worker.py:229-256]

**MEDIUM-10.10-3 — Test `test_worker_skip_locked_...` ne valide pas le multi-process réel**

`asyncio.gather(process_outbox_batch(engine), process_outbox_batch(engine))` → 2 coroutines dans **le même event loop**, partageant **le même connection pool**. Le verrou `FOR UPDATE SKIP LOCKED` est testé sur des sessions différentes (`async with AsyncSession(engine) as db` 2×) mais issues du même pool, même process.

Le scénario prod (2 workers gunicorn × N replicas) n'est **pas couvert** : 2 connexions PostgreSQL réelles vers le même backend via 2 process distincts. Le pattern reste correct par construction PostgreSQL, mais le test donne une **fausse confiance**.

**Remédiation Epic 20 Ops** : test avec `multiprocessing.Pool` + 2 engines indépendants (chaque process `create_async_engine`), OU test docker-compose 2 conteneurs FastAPI. Flaggé LOW dans le brief challenger, mais la fausse confiance le hisse MEDIUM.

### 🔹 Observations mineures (LOW / INFO)

**LOW-10.10-1 — `scheduler.shutdown(wait=True)` bloque l'event loop**
APScheduler `AsyncIOScheduler.shutdown()` est une méthode **synchrone** qui attend les jobs en cours. Appelée depuis `stop_outbox_scheduler` (`async def`), elle bloque l'event loop pendant la finalisation (potentiellement plusieurs secondes si un batch de 100 events est en cours). Acceptable au shutdown FastAPI (moment unique, pas de trafic concurrent). [worker.py:310-315]

**LOW-10.10-2 — `db=None` toléré en tests handlers**
`test_handlers.py` et `test_log_structure.py` passent `db=None` à `dispatch_event` / `_process_single_event`. Fonctionne car `noop_handler` et le chemin `unknown_handler` ne touchent pas `db`. Fragile si un test futur ajoute un handler SQL et oublie de fournir une vraie session. Pragma acceptable, à surveiller Epic 13.

**LOW-10.10-3 — `ck_domain_events_status CHECK status IN (...)` absent**
Déjà tracé dans `deferred-work.md` comme dette LOW née de Story 10.10 (Q3). Confirmé acceptable MVP (4 valeurs assurées par le code applicatif). [models/domain_event.py, deferred-work.md §Story 10.10]

**LOW-10.10-4 — Redondance `status == "pending"` + `processed_at IS NULL`**
Les 4 états finaux (`processed` / `failed` / `unknown_handler`) mettent `processed_at` non-NULL. La clause `processed_at IS NULL` implique donc `status == "pending"`. Laisser les deux est défense-en-profondeur (ok). Micro-optimisation possible (retirer le doublon), non requis. [worker.py:110-112]

**LOW-10.10-5 — `purge_expired_prefill_drafts` en raw SQL vs ORM**
Pas de modèle `PrefillDraft` ORM encore livré → raw `text()` acceptable. À harmoniser Epic 16 (quand PrefillDraft ORM sera livré pour la Story 16.5 copilot deep-link). [worker.py:238-248]

**INFO-10.10-1 — `noop.test` handler en registre production**
Test-only documenté, acceptable MVP. Candidat Epic 20 Ops : skipper l'enregistrement si `env_name == "prod"` (réduire surface d'attaque). [handlers.py:82-118]

**INFO-10.10-2 — Scan CI unicité writer = 2 hits (attendu 1)**
`rg "INSERT INTO domain_events|DomainEvent\("` matche `writer.py:95` **ET** `models/domain_event.py` (référence ORM via la docstring `DomainEvent(id=..., ...)` et le docstring d'`__repr__`). Scan en prod devrait exclure `backend/app/models/*` pour garantir 1 hit strict. Correction trivial (glob exclude).

**INFO-10.10-3 — 2 schedulers dans même process impossibles** — confirmé : un seul `AsyncIOScheduler` instancié au lifespan (`main.py:41`). Point §5 Pièges #8 respecté. ✓

**INFO-10.10-4 — Registry import-time reproducibility** — `_validate_unique_event_types()` exécuté une fois à l'import. Tests qui `monkeypatch.setattr(handlers_mod, "EVENT_HANDLERS", ...)` + re-invoque le validator sont idempotents (scan linéaire, pas de cache). ✓

**INFO-10.10-5 — Timezone end-to-end UTC** — scheduler, `_utcnow()`, `TIMESTAMPTZ`, logs `isoformat()` avec offset. Aucun `datetime.now()` naïf en prod. ✓
