### [HIGH-10.5-1] AC4 test matrix ne vérifie PAS la chaîne « bypass admin → ligne `admin_access_audit` écrite »

**Fichier** : `backend/tests/test_security/test_rls_enforcement.py:214-254`
**AC référencé** : AC4 — texte littéral :
> Pour chaque scénario `admin_mefali` / `admin_super`, [...] le même SELECT retourne **≥ 1 ligne**, **et** une ligne `admin_access_audit` existe avec `table_accessed=<table>`, `admin_role=<role>`, `admin_user_id=<admin.id>`. (Note : le listener AC3 est `before_flush` — donc ce test ajoute explicitement une **opération mutante** (ex. `UPDATE companies SET name=name WHERE id=... ` no-op) pour déclencher le flush et vérifier l'audit.)

**Constat** : le test livré `test_rls_matrix` fait un `SELECT id FROM <table> WHERE id = :id` via `sync_engine.connect()` + `SET LOCAL ROLE app_user`. Il vérifie `assert row is not None` pour les admin roles — ce qui prouve le **bypass RLS** mais **pas** l'**enregistrement par le listener**. Trois problèmes imbriqués :

1. **Aucune mutation** : le test ne fait qu'un SELECT → `before_flush` ne se déclenche pas → aucune insertion `admin_access_audit` n'est attendue ni vérifiée.
2. **Mauvais engine pour le listener** : le listener est attaché sur `AsyncSession.sync_session_class` (`event.listens_for(AsyncSession.sync_session_class, "before_flush")` — `admin_audit_listener.py:108`). Or `sync_engine` des tests est un `create_engine(...)` psycopg2 direct qui **ne passe pas par `AsyncSession`** — donc même si le test ajoutait une mutation, le listener ne serait pas attaché à ces sessions. Pour tester l'écriture audit, il faut soit instancier un `async_session_factory` + `register_admin_access_listener(engine)`, soit un test d'intégration FastAPI via `httpx.AsyncClient + get_db` override.
3. **Silence sur la limitation** : aucune assertion ne vérifie la **présence** ET aucun commentaire ne documente que le test intentionnellement skip la chaîne listener. La couverture AC4 « ligne `admin_access_audit` existe » est donc **absente de la suite**.

**Risque opérationnel** : le listener `before_flush` pourrait être **cassé** (ex. quelqu'un modifie le filtre `RLS_TABLE_NAMES`, retire l'appel `register_admin_access_listener` du `lifespan` de `main.py`, ou brise `AsyncSession.sync_session_class` lors d'un upgrade SQLAlchemy) **sans aucune régression détectée par la suite test Story 10.5**. La conformité D7 (« tout bypass admin doit être logué ») reposerait alors sur un code non testé de bout en bout.

**Preuve factuelle** : aucun test dans `backend/tests/test_security/` ne fait un `SELECT ... FROM admin_access_audit WHERE table_accessed = ...` après une mutation admin. Grep confirme :

```bash
rg -n "FROM admin_access_audit" backend/tests/test_security/
# test_audit_tamper_proof.py:46 (seed), 106 (read-back), 112, 131, 150 — jamais dans un flux listener
```

**Recommandation (choisir 1)** :

- **Option A (recommandée, ~40 lignes)** — ajouter un test paramétré `test_listener_writes_audit_entry_on_admin_mutation` dans `test_rls_enforcement.py` :

  ```python

  from app.core.admin_audit_listener import register_admin_access_listener
  from app.core.rls import apply_rls_context

  @pytest.mark.parametrize("role,table", [
      ("admin_mefali", "companies"),
      ("admin_super", "fund_applications"),
      ("admin_super", "facts"),
      ("admin_mefali", "documents"),

  async def test_listener_writes_audit_entry_on_admin_mutation(
      alembic_postgres_url, rls_fixture, monkeypatch, role, table

      monkeypatch.setenv(

      engine = create_async_engine(_to_async_url(alembic_postgres_url))
      register_admin_access_listener(engine)
      factory = async_sessionmaker(engine, expire_on_commit=False)
      admin = _fake_user(uuid.uuid4(), "bypass@mefali.com")
      async with factory() as session:
          await apply_rls_context(session, admin)
          # Mutation no-op : UPDATE d'une ligne appartenant à un autre tenant
          target_id = rls_fixture[table]["b"]
          await session.execute(
              text(f"UPDATE {table} SET updated_at = now() WHERE id = :id"),
              {"id": target_id},
          )
          await session.commit()
          logs = (await session.execute(
              text("SELECT table_accessed, admin_role, admin_user_id "
  ```

### [MEDIUM-10.5-1] `record_ids` JSON construit par f-string (risque injection théorique)

**Fichier** : `backend/app/core/admin_audit_listener.py:161-163`

**Constat** :

```python
record_id_raw = getattr(obj, "id", None)
record_ids_json = (
    f'["{record_id_raw}"]' if record_id_raw is not None else "[]"
)
```

Le JSON est construit par f-string à partir de `obj.id`, qui est aujourd'hui un UUID (format sûr). **Mais** :

1. Le listener a une portée volontairement large (toute table mutée avec `__tablename__ in RLS_TABLE_NAMES`). Si un jour une de ces 4 tables utilise une PK composite ou une clé business arbitraire (string `"order-2026-001"`, ou pire un slug user-contrôlable), l'injection JSON devient triviale : `id = 'x"},{"admin":"backdoor'` produit un JSON qui passe `CAST(... AS JSONB)` avec un **payload injecté**, ce qui peut corrompre des queries ultérieures de l'UI d'audit.
2. Defense in depth contradictoire : `apply_rls_context` utilise **bind params** précisément pour cette raison (commenté explicitement — « défense en profondeur contre toute injection future si `user.id` devenait partiellement contrôlable »). Le listener **ne respecte pas** ce principe.
3. L'effort est minimal — Python standard lib `json.dumps` gère nativement le quoting.

**Recommandation** :

```python
import json
record_ids_json = json.dumps([str(record_id_raw)] if record_id_raw is not None else [])
```

Garde le binding existant avec `{"record_ids": record_ids_json}` + `CAST(:record_ids AS JSONB)`. Aucun changement de schéma, zéro test à ajouter.

### [MEDIUM-10.5-2] Docstring `get_db` suggère que `pool_pre_ping=True` protège des fuites de contexte RLS — **faux**

**Fichier** : `backend/app/core/database.py:40-48`

**Constat** : le commentaire inline affirme :

> *la connexion sera de toute façon remise au pool dans un état invalide et rejetée au prochain usage par `pool_pre_ping=True`*

C'est **incorrect**. `pool_pre_ping=True` exécute un `SELECT 1` avant livraison d'une connexion du pool. Il vérifie que la connexion TCP est **viable**, pas que ses settings de session (`app.current_user_id`, `app.user_role`) sont propres. Une connexion laissée avec `app.current_user_id='uuid-A'` suite à un échec de `reset_rls_context` sera remise au pool, survivra au `pre_ping`, et **délivrée telle quelle** à une requête suivante.

**Mitigation effective** : chaque requête authentifiée passe par `get_current_user → apply_rls_context(db, user)` qui **écrase** le setting avant toute query. Le risque réel est borné aux cas extrêmement rares où un endpoint accède aux 4 tables RLS **sans** `Depends(get_current_user)` (aucun dans le code actuel). Le fail-safe tient donc, mais **pas pour la raison indiquée**.

**Recommandation** : remplacer le commentaire par :

```python
# Story 10.5 — reset défensif du contexte RLS.
# Dérogation C1 (leçon 9.7) justifiée : si la session est déjà invalide
# (rollback sur erreur métier), propager l'exception de reset masquerait
# l'erreur réelle ; le reset n'a que valeur défensive (belt-and-braces).
# NOTE sécurité : `pool_pre_ping=True` NE protège PAS contre un contexte
# RLS résiduel (il ne fait qu'un SELECT 1). La vraie garantie est
# qu'apply_rls_context écrase systématiquement les settings au début de
# chaque requête authentifiée (cf. get_current_user).
with contextlib.suppress(Exception):
    await reset_rls_context(session)
```

## Layer 2 — SÉCURITÉ CRITIQUE

### [LOW-10.5-1] Limitation « un user PostgreSQL superuser bypasse REVOKE + trigger » non documentée dans `security-rls.md §6`

**Fichier** : `docs/CODEMAPS/security-rls.md:182-199`

**Constat** : la section « Limitations connues » liste 4 points (SELECT non auditable, whitelist transitoire, rôle auditor manquant, rappel SQL injection). Elle **ne documente pas** deux vecteurs d'attaque connus de RLS/triggers PostgreSQL :

1. **SUPERUSER PG bypasse les triggers plpgsql simples** : un compte avec l'attribut SUPERUSER ne voit PAS le trigger `audit_table_is_immutable` se déclencher pour les UPDATE/DELETE si le trigger est défini comme `BEFORE ... FOR EACH ROW` (il se déclenche, mais un SUPERUSER peut le désactiver via `ALTER TABLE admin_access_audit DISABLE TRIGGER trg_admin_access_audit_immutable`). Correction technique : le trigger plpgsql **SE déclenche** pour un SUPERUSER (les triggers normaux s'appliquent à tous sauf propriétaire de table en mode `DISABLE TRIGGER`), mais un SUPERUSER peut les **désactiver** à volonté.
2. **REVOKE sur PUBLIC ne s'applique pas à l'owner de la table ni aux SUPERUSER** : donc le user `postgres` admin de la base peut bypasser REVOKE directement.

Ces deux points sont **connus et acceptables** (on fait confiance à l'ops qui gère les credentials SUPERUSER), mais doivent être **explicités** dans la section limitations pour éviter un faux sentiment de sécurité en revue SOC2/ISO. Architecturalement, c'est pourquoi D7 exige aussi la ségrégation stricte des accès opérationnels à PostgreSQL (à livrer plus tard).

**Recommandation** : ajouter une 5ᵉ ligne au tableau de `security-rls.md §6` :

```
| 5 | **SUPERUSER PG bypasse REVOKE + trigger** (attribut rolsuper + `DISABLE TRIGGER`) | Les protections Story 10.5 reposent sur l'hypothèse que les accès opérationnels PostgreSQL sont ségrégués | Épic 20 Ops — role séparé `app_readonly`/`app_write` sans SUPERUSER, audit des accès SUPERUSER au niveau infra (pg_log + Cloud SQL audit) |
```

+ ajouter en tête du §5.2 (« Trigger `BEFORE UPDATE OR DELETE` ») la note : « Un user PG avec attribut SUPERUSER ou propriétaire de table peut désactiver ce trigger via `DISABLE TRIGGER`. La ségrégation des credentials prod est hors scope 10.5 (cf. §6.5). »

### [LOW-10.5-2] Aucun test ne valide le REVOKE isolément (seul le trigger est testé)

**Fichier** : `backend/tests/test_security/test_audit_tamper_proof.py`

**Constat** : les 6 tests reposent sur un `sync_engine` connecté en superuser de migration. Le `REVOKE UPDATE, DELETE ON admin_access_audit FROM PUBLIC` **n'empêche rien** pour ce superuser ; seul le `TRIGGER` bloque. Conséquence :

- Si demain quelqu'un retire accidentellement le bloc `CREATE TRIGGER` de la migration 028 en laissant le REVOKE, **les 6 tests passeraient** (en superuser, REVOKE n'a pas d'effet, et sans trigger il n'y a plus de blocage).
- Aucune assertion ne teste que **l'utilisateur applicatif effectif** (ex. `app_user` non-superuser) est bloqué par le REVOKE seul.

**Risque réel** : faible (scénario régressif très spécifique). **Gap de défense en profondeur** : oui.

**Recommandation (optionnelle, déférable)** : ajouter `test_revoke_blocks_non_superuser_update` utilisant `SET LOCAL ROLE app_user` (pattern déjà établi dans `test_rls_enforcement.py:193-206`) :

```python
def test_revoke_update_on_admin_access_audit_blocks_non_superuser(sync_engine):
    """AC6 défense indépendante : REVOKE bloque UPDATE même si trigger inactif."""
    with sync_engine.begin() as conn:
        _ensure_app_user_role(conn)  # helper de test_rls_enforcement.py
        admin_id = _seed_user(conn)
        log_id = _seed_admin_access_audit(conn, admin_id)
    with pytest.raises(ProgrammingError) as exc_info:
        with sync_engine.begin() as conn:
            conn.execute(text("SET LOCAL ROLE app_user"))
            conn.execute(text("UPDATE admin_access_audit SET reason='x' WHERE id = :id"), {"id": log_id})
    # Priorité au REVOKE : "permission denied" AVANT le trigger.
    assert "permission denied" in str(exc_info.value).lower() \
        or "audit table is immutable" in str(exc_info.value).lower()
```

Accepté si déféré : ajouter item `deferred-work.md` explicite.

### [LOW-10.5-3] `ERRCODE = '42501'` (insufficient_privilege) choisi — sémantiquement ambigu

**Fichier** : `backend/alembic/versions/028_tamper_proof_audit_tables.py:53-54`

**Constat** : le code `42501` signifie *insufficient privilege* dans SQLSTATE. Ici l'exception est levée par un trigger PL/pgSQL, pas par le système de privilèges PG. Un observateur (SIEM, outil d'alerte) pourrait confondre avec un véritable `permission denied` (ex. REVOKE non-superuser). Pas de bug, mais **dilution du signal** en cas d'investigation.

Alternative envisageable : `'P0001'` (raise_exception générique) ou `'P0002'` (no_data_found) ou code custom préfixé `ME` (classe *Mefali*). PostgreSQL autorise les codes custom en classe 'XX' ou en préfixe défini par l'application.

**Recommandation (INFO, déférable)** : si `42501` est conservé, documenter le choix dans `security-rls.md §5.2` (« choix : 42501 pour homogénéiser avec permission_denied upstream applicatif, au prix d'une ambiguïté SIEM — à arbitrer Epic 20 Ops avec l'équipe SRE »).

### [LOW-10.5-4] Le `RESET ROLE` explicite dans `test_rls_matrix:245` est redondant avec la fin de transaction

**Fichier** : `backend/tests/test_security/test_rls_enforcement.py:245`

**Constat** : `SET LOCAL ROLE app_user` est automatiquement réinitialisé à la fin de la transaction par PostgreSQL. ### [LOW-10.5-5] `apply_rls_context` : l'ordre des 2 `db.execute(set_config)` n'est pas atomique

**Fichier** : `backend/app/core/rls.py:131-138`

**Constat** : deux appels `set_config` séquentiels, non encapsulés dans une même transaction explicite. Dans le cas pathologique où la 1ère query réussit puis la 2ᵉ échoue (ex. perte réseau transient), la session serait dans l'état incohérent `(user_id=<valide>, role=<stale de la requête précédente>)`. Si la requête précédente avait role=`admin_super`, l'utilisateur courant hériterait d'un bypass non désiré jusqu'au prochain `apply_rls_context` / `reset`.

**Mitigation existante** : `reset_rls_context` dans `get_db.finally` nettoie l'état au retour au pool. Si la requête suivante est authentifiée, `apply_rls_context` est appelé en premier et écrase. Risque effectif : très faible (transience + chemin non-authentifié simultanés).

**Recommandation (INFO, déférable)** : remplacer par un seul appel combiné pour atomicité :

```python
await db.execute(
    text(
        "SELECT set_config('app.current_user_id', :uid, false), "
        "       set_config('app.user_role', :role, false)"
    ),
    {"uid": user_id_value, "role": role_value},
)
```

Gain : 1 roundtrip réseau au lieu de 2, + atomicité. Zéro changement sémantique.

## Challenges spécifiques 10.5 — réponses

### AC6 « REVOKE + trigger = double protection ? Que se passe-t-il si pg_hba.conf permet un user d'outrepasser REVOKE ? »

- **REVOKE sur PUBLIC** ne bloque pas un user avec attribut SUPERUSER ni le propriétaire de la table. `pg_hba.conf` règle l'authentification (qui peut se connecter), pas les privilèges objet — il ne peut donc pas outrepasser REVOKE directement. Mais il peut autoriser un login SUPERUSER (ex. `trust` en local dev) qui bypasse tout. **Couvert par LOW-10.5-1** : à documenter en limitations section 6 de security-rls.md.
- **Trigger** : s'exécute pour tout user y compris SUPERUSER, mais peut être désactivé par SUPERUSER via `DISABLE TRIGGER ALL`. Défense supplémentaire mais pas absolue — d'où l'importance de la ségrégation ops prod (hors scope 10.5).

### AC3 « listener before_flush n'intercepte pas les SELECT admin (limitation documentée ?) »

✅ Documentée 2 fois :
- `admin_audit_listener.py:10-16` (docstring module)
- `docs/CODEMAPS/security-rls.md §4` (Limitation SELECT — citation textuelle) + §6 item 1 (roadmap Story 18.x)

### AC5 « pool-safe reset : que se passe-t-il si une transaction commit avant reset ? »

✅ Comportement correct par design : `set_config(..., false)` a scope session (persiste après commit). Le `reset_rls_context` post-commit ré-exécute le `set_config` dans une nouvelle transaction implicite (ou pas — SQLAlchemy/asyncpg gère l'auto-begin). Même si la nouvelle transaction ne commit jamais (session fermée), le setting `is_local=false` persiste jusqu'à fermeture de la connexion physique. **Tests `test_session_reset_clears_rls_context` + `test_sequential_requests_isolation` valident la chaîne complète apply → commit implicite → reset → check empty → pool reuse check empty.**

### AC6 « Migration 028 idempotente : re-run downgrade+upgrade ne casse pas ? »

✅ Upgrade idempotent via `CREATE OR REPLACE FUNCTION` + `DO $$ IF NOT EXISTS ... END $$`. Downgrade idempotent via `DROP TRIGGER IF EXISTS` + `DROP FUNCTION IF EXISTS` + `GRANT` (PG autorise GRANT même si déjà détenu). Pattern vérifiable localement via `alembic downgrade 027_outbox_prefill && alembic upgrade head && alembic downgrade 027_outbox_prefill && alembic upgrade head` — sans erreur.

### AC5 « Tests E2E : admin_super tente UPDATE admin_access_audit → échec certifié ? »

✅ Littéralement : `test_update_admin_access_audit_raises_immutable` + `test_delete_admin_access_audit_raises_immutable` (× 2 tables = 4 tests UPDATE/DELETE). Assertion : le message d'exception contient `"audit table is immutable"` (case-insensitive). Satisfaisant. Complément recommandé (LOW-10.5-2) : tester le REVOKE isolément en non-superuser.

### Bypass trigger via superuser PG — documenté en §limits ?

⚠️ Non documenté dans `security-rls.md §6`. **Couvert par LOW-10.5-1.**

### ORM vs SQL direct — listener couvre-t-il `db.execute(text("DELETE..."))` ?

**Non.** `before_flush` se déclenche uniquement sur flush ORM, pas sur `session.execute(text(...))` brut. Un admin exécutant `await session.execute(text("DELETE FROM companies WHERE owner_user_id = :other_tenant"), ...)` ne serait **pas** audité. **Limitation importante non explicitement documentée** (la doc mentionne seulement la limitation SELECT).

**Recommandation (non-bloquant, à considérer)** : documenter dans `security-rls.md §4 Contrat du listener admin` — ajouter un bullet :

> **Le listener ne capture PAS** :
> - Les SELECT (limitation intrinsèque `before_flush`, cf. Limitation 1).
> - Les requêtes SQL brutes via `session.execute(text("UPDATE/DELETE ..."))` — seul l'ORM (`session.add`, mutation d'attribut sur objet mappé, `session.delete`) déclenche `before_flush`. Les stores/services applicatifs doivent privilégier l'ORM pour les 4 tables RLS, ou compléter manuellement avec un log explicite `admin_access_audit`.

À considérer aussi comme INFO-10.5-4, dépendant de l'appréciation du Product Owner. Le risque pratique est faible : le codebase `backend/app/modules/` utilise massivement l'ORM (sessions scoped, `.add()` + `.commit()`) ; les rares `text(...)` concernent les migrations ou les tests.

### admin_access_audit + admin_catalogue_audit_trail protection symétrique ?

✅ Migration 028 ligne 33 : `AUDIT_TABLES = ("admin_access_audit", "admin_catalogue_audit_trail")` — itéré pour REVOKE + trigger. Tests ligne 144-191 : 3 tests par table (INSERT ok, UPDATE raise, DELETE raise). **Symétrie confirmée.**

### Enum admin_role_enum strict (pas de fallback accidentel) ?

✅ Enum PostgreSQL `admin_role_enum {admin_mefali, admin_super}` créé par migration 024. Le listener filtre `if role not in ADMIN_ROLES: return` avant l'INSERT, donc aucune valeur hors enum ne peut atteindre la table. Si demain quelqu'un élargit `ADMIN_ROLES` sans élargir l'enum, le listener échouera avec `ProgrammingError` — **fail-loud, pas fail-silent** (comportement souhaité).

### set_config session-scoped — fuite inter-transactions ?

✅ Le scope est bien session (pas transaction) — c'est intentionnel (survivre aux commits intermédiaires du même cycle request). La protection est double :
- `reset_rls_context` en `finally` de `get_db` (au retour au pool)
- `apply_rls_context` en tête de `get_current_user` (écrasement garanti à chaque nouvelle requête authentifiée)

Le test `test_sequential_requests_isolation` prouve la chaîne complète.
