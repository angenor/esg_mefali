### CRITICAL-1 — Corpus golden recall@5 : 10 chunks réels au lieu de 50 + ground-truth inexistants

**Location** : `backend/tests/test_core/fixtures/embeddings_quality_corpus.json`, `backend/tests/test_core/test_embeddings_quality.py:44-51`

**Évidence** :
- AC6 spec : "15 queries + **50 chunks golden** pré-embeddés avec `expected_top5_chunk_ids` ground-truth".
- Fixture JSON livrée contient **10 chunks** : `c-01, c-06, c-11, c-16, c-21, c-26, c-31, c-36, c-41, c-46`.
- Queries référencent `c-01..c-05`, `c-06..c-10`, …, `c-46..c-50` dans leurs `expected_top5_chunk_ids` — soit **40 chunks ID référencés qui n'existent pas** dans le corpus.
- `test_corpus_structure_is_valid:48` assert `len(chunks) >= 10` (pas `== 50`) — **masque silencieusement la lacune**.
- Les 5 queries EN EUDR (`q-eudr-en-01..05`) pointent **toutes** sur les mêmes 5 chunks attendus `[c-41..c-45]` — zéro diversité ground-truth. Et seul `c-41` existe.

**Conséquence** :
- `_recall_at_5(retrieved, expected)` = `|set(retrieved) ∩ set(expected)| / 5`. Pour chaque query, max 1 chunk attendu existe → recall@5 théorique plafonné à 0.2.
- La gate `voyage_mean ≥ openai_mean - 0.05` est **symbolique** : les deux providers buteront sur les chunks inexistants → comparaison relative correcte si les deux sont déterministes, mais la qualité absolue (AC6 implicit) n'est pas mesurée.
- Le test gated network (`EMBEDDINGS_QUALITY_CHECK=1`) passe sur un corpus biaisé sans signal fiable : un Voyage qui retourne les chunks **inverses** des attentes passerait toujours si OpenAI est symétriquement mauvais.

**Piège 7 documenté** ("recall@5 corpus de référence biais", story §14.7) avertissait "biais possible : chunks golden sélectionnés avec un mental model proche de voyage-3". Ici le biais est bien plus grave : **corpus incomplet** masqué par un test structurel lâche.

**Triage** : `patch` (2 options fix).

### HIGH-4 — Classification d'exceptions par substring dans `openai.py`

**Location** : `backend/app/core/embeddings/openai.py:86-96`

**Évidence** :
```python
try:
    vectors = await client.aembed_documents(texts)
except Exception as exc:  # noqa: BLE001 — wrap en canoniques
    msg = str(exc).lower()
    if "rate" in msg or "429" in msg:
        raise EmbeddingRateLimitError(str(exc)) from exc
    raise EmbeddingError(f"OpenAI embedding failed: {exc}") from exc
```

- Violation C1 leçon 9.7. Le test `test_no_generic_except_in_embeddings_module:262-285` a été **volontairement lâché** pour accepter ce hit : `assert len(offenders) <= 1`. Commentaire test : "openai.py a 1 except Exception contrôlé (mapping canonique) — toléré".
- La classification `msg.lower()` substring match `"rate" in msg or "429" in msg` est fragile :
  - Exception FR "Trop de requêtes" (certains providers localisés) → non détectée → `EmbeddingError` générique → `_breaker.record_failure(is_llm_5xx=True)` OUI via chemin `voyage.py:181` mais **voyageai n'est pas appelé ici** ; dans `OpenAIEmbeddingProvider` standalone, aucun breaker n'est branché, donc `EmbeddingError` propagé à `store_embeddings` ligne 550 → `embeddings = [None] * len(chunks)` → silent data loss.
  - `openai.RateLimitError`, `openai.APIError`, `httpx.HTTPStatusError` sont des classes propres : l'idiome Python correct est `except openai.RateLimitError: ... except (openai.APIError, httpx.HTTPError): ...` — pas string match.

**Triage** : `patch`.

```python
# Suggested fix (openai.py:86-96)
try:
    vectors = await client.aembed_documents(texts)
except Exception as exc:
    # Import lazy pour éviter hard-dep openai brut.
    try:
        import openai  # type: ignore[import]
        if isinstance(exc, openai.RateLimitError):
            raise EmbeddingRateLimitError(str(exc)) from exc
        if isinstance(exc, openai.APIError):
            raise EmbeddingError(f"OpenAI API error: {exc}") from exc
    except ImportError:
        pass
    raise EmbeddingError(f"OpenAI embedding failed: {exc}") from exc
```

Ou mieux : catcher `openai.RateLimitError` / `openai.APIError` en `except` séparés et supprimer le `except Exception` → 0 hit.
