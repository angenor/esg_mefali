### HIGH-10.12-2 — Log méta-audit `audit_export_issued` perdu si client disconnect

**Fichier** : `backend/app/modules/admin_catalogue/router.py:484-517`

**Symptôme** : le log `logger.info("audit_export_issued", extra={...})` est émis à la fin du générateur `_stream_rows`, APRÈS la boucle `async for row in result.scalars()`. Si le client HTTP se déconnecte en cours de streaming (Ctrl+C navigateur, timeout reverse-proxy, perte réseau), Starlette/FastAPI annule le générateur via `asyncio.CancelledError` et le `logger.info` final n'est jamais exécuté.

**Pourquoi c'est un HIGH** :
- Un admin compromis peut déclencher un export massif, commencer à recevoir les chunks (les premiers Ko suffisent si l'attaque vise des rows récents en tête `ORDER BY ts DESC`), puis couper la connexion. Aucun log `audit_export_issued` n'est produit.
- C'est **exactement le cas d'usage** pour lequel le log méta-audit a été introduit (défense en profondeur « un admin qui exporte est lui-même audité », voir story ligne 62).
- Le checklist utilisateur demande explicitement « RLS admin bypass listener Story 10.5 : consultation GET audit-trail déclenche listener avant_flush ? audit log de l'accès audit ? (méta-audit) » — la réponse actuelle est « partiellement, et perdu sous conditions d'attaque ».

**Remédiation** :
```python
async def _stream_rows():
    # Log d'intention AVANT le premier yield (survit à disconnect)
    logger.info(
        "audit_export_started",
        extra={
            "actor_user_id": str(admin.id),
            "filters_hash": filters_hash,
        },
    )
    count = 0
    truncated = False
    try:
        # ... boucle existante ...
        async for row in result.scalars():
            # ...
    finally:
        # Log de fin toujours émis (complet OU interrompu)
        logger.info(
            "audit_export_completed" if not truncated else "audit_export_truncated",
            extra={
                "actor_user_id": str(admin.id),
                "filters_hash": filters_hash,
                "row_count": count,
                "truncated": truncated,
            },
        )
```

Coût : ~5 min, 8 lignes.

**Source** : `backend/app/modules/admin_catalogue/router.py:484-517`

### HIGH-10.12-3 — Double query + race window + OOM risk sur export CSV

**Fichier** : `backend/app/modules/admin_catalogue/router.py:472-540`

**Symptôme** : l'endpoint `export_audit_trail_csv` exécute deux queries distinctes :

1. **Pre-query** (lignes 532-540) : `SELECT id FROM admin_catalogue_audit_trail WHERE <clauses> LIMIT 50_001` pour détecter la troncature et poser `X-Export-Truncated: true` dans les headers AVANT le premier chunk.
2. **Streaming query** (lignes 472-480) : `select(...).execution_options(yield_per=500)` via `db.stream()` pour produire le CSV.

**3 problèmes** :

1. **Coût I/O doublé** : chaque export scanne la table deux fois (même index, mais deux allers-retours).
2. **Race window** : entre la pre-query et le début du streaming, une insertion concurrente (nouvelle mutation catalogue Epic 13) peut faire passer le compte de 49 999 → 50 001. Le header dirait `X-Export-Truncated: false` mais le stream s'arrêterait à la sentinelle. Le client reçoit un signal contradictoire.
3. **Anti-pattern streaming** : `count_result.scalars().all()` (ligne 538) ramène **jusqu'à 50 001 UUIDs** en mémoire Python (~800 Ko RAM) avant le premier yield. Défait l'objectif du streaming (mémoire bornée à 500 rows × ~200 bytes = 100 Ko).

**Pourquoi c'est un HIGH** : le design annoncé dans la story ligne 158 est « `yield_per=500` permet de streamer 50k rows en ≤ 30 s sans OOM (mémoire bornée au batch 500) » — la pre-query contredit cet invariant. Sur une table Phase 3 (>500k rows total, filtre admin retourne 100k), c'est 100 001 UUIDs en RAM.

**Remédiation** :
```python
# Option MVP (simple) : supprimer le header X-Export-Truncated côté HTTP
# et laisser la sentinelle CSV faire foi (déjà présente ligne 498).
# Le header est cosmétique — la sentinelle `# TRUNCATED_AT_50000_ROWS` est
# le contrat client documenté (§Consultation audit-trail.md).

# Supprimer lignes 524-540 (pré-query + test len) — garder uniquement :
return StreamingResponse(
    _stream_rows(),
    media_type="text/csv; charset=utf-8",
    headers={
        "Content-Disposition": f'attachment; filename="{filename}"',
    },
)
```

Le client JS/curl détecte la troncature en scannant la dernière ligne du CSV pour la sentinelle — c'est déjà le contrat documenté. Le header HTTP `X-Export-Truncated` était un nice-to-have, pas un AC.

Alternative si le header est requis : retenir le dernier batch avant yield et logger `truncated` en trailing header HTTP (`Trailer:` + `Trailer-Values`) — complexité > bénéfice, préférer suppression.

Coût : ~5 min, -10 lignes.

**Source** : `backend/app/modules/admin_catalogue/router.py:519-540`

### MEDIUM-10.12-1 — Cursor base64 non-signé (MVP acceptable, à documenter dans §Pièges)

**Analyse** : le cursor `base64(f"{ts_iso}|{uuid_hex}")` est opaque mais **pas** HMAC-signé. Un admin authentifié peut forger trivialement un cursor arbitraire.

**Impact réel (investigué)** : le filtre est `WHERE (ts, id) < (cursor_ts, cursor_id)`. Un cursor forgé vers le futur (`9999-12-31T23:59:59|ffffffff-...`) retournerait toutes les rows (comportement identique à une requête sans cursor). Un cursor vers le passé retournerait moins de rows. **Aucun bypass d'autorisation** : l'admin a déjà accès à toutes les rows via pagination normale.

**Conclusion** : pas de vulnérabilité sécurité. Acceptable MVP. Mais le checklist utilisateur demande explicitement « non tamper-able (signature HMAC OU validation stricte) » — l'absence est à documenter pour éviter un futur reviewer de le flagger CRITICAL sans l'analyse ci-dessus.

**Remédiation** : ajouter un bullet dans `docs/CODEMAPS/audit-trail.md §Pièges` :
> **Cursor opaque non-signé** : le cursor est base64 trivialement forgeable, mais sans impact sécurité. Le filtre `(ts, id) < cursor` limite à **réduire** le jeu de résultats — un cursor forgé donne au pire ce que renvoie une requête sans cursor. Pas d'escalade d'autorisation. HMAC-signé rejeté MVP (pattern industry OK, GitHub/Stripe/Slack n'HMAC pas leurs cursors non plus).

Coût : ~2 min, 4 lignes doc.

### MEDIUM-10.12-2 — Test pagination ne vérifie pas la disjonction page1 ∩ page2 = ∅

**Fichier** : `backend/tests/test_admin_catalogue/test_audit_trail_endpoint.py:166-173`

**Symptôme** :
```python
page1_ids = {item["id"] for item in body1["items"]}
page2_ids = {item["id"] for item in body2["items"]}
all_seed_ids = {str(row.id) for row in seeded_audit_rows}
assert (page1_ids | page2_ids) == all_seed_ids, (
    f"page1={page1_ids} page2={page2_ids} seeds={all_seed_ids}"
)
```

L'assertion vérifie que l'**union** page1 ∪ page2 == seeds, mais pas l'**intersection** vide. Si un bug de cursor utilisait `<=` au lieu de `<` strict, un doublon apparaîtrait sur la dernière row de page1 / première row de page2 — l'union masquerait le doublon.

**Remédiation** : ajouter 1 ligne
```python
assert page1_ids.isdisjoint(page2_ids), (
    f"Duplicate rows across pages : {page1_ids & page2_ids}"
)
```

Coût : ~1 min, 3 lignes.

### LOW-10.12-1 — `_FORMULA_INJECTION_PREFIXES` sans `\n`

**Fichier** : `backend/app/modules/admin_catalogue/router.py:145`

**Symptôme** : `_FORMULA_INJECTION_PREFIXES = ("=", "+", "-", "@", "\t", "\r")` — manque `\n`.

**Impact** : CSV injection via `\n` seul (sans `\r`) reste possible dans certains parseurs CSV tolérants (LibreOffice Calc < 7.2, certains parsers Python tiers). OWASP CSV Injection recommande `\t\r\n` comme triplet complet.

**Remédiation** : `_FORMULA_INJECTION_PREFIXES = ("=", "+", "-", "@", "\t", "\r", "\n")` + ajouter un test parametrize pour `\n`.

Coût : ~2 min.
