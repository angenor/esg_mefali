## Layer 1 — Code Quality Frontend

### HIGH

**HIGH-1 · `test_button_types.ts` ne vérifie PAS le compile-time enforcement AC5**
Le fichier `frontend/tests/components/ui/test_button_types.ts` prétend (docstring ligne 5) utiliser des directives `// @ts-expect-error` pour attester que le type discriminé rejette `<Button iconOnly />` sans `ariaLabel`. Dans les faits, le fichier livré ne contient QUE des type-aliases d'assertion sur les unions :

```ts
type _AssertVariants = ButtonVariant extends 'primary' | 'secondary' | 'ghost' | 'danger'
  ? true
  : never;
type _AssertSizes = ButtonSize extends 'sm' | 'md' | 'lg' ? true : never;
```

Ces assertions vérifient que `ButtonVariant` et `ButtonSize` ont les membres attendus, **pas** que le type discriminé refuse l'usage invalide. Aucune directive `// @ts-expect-error` n'apparaît dans le fichier (`grep -n "ts-expect-error"` retourne uniquement la ligne de docstring).

**Risque** : l'enforcement compile-time AC5 repose sur `Button.vue:20-22` mais n'est **pas testé**. Un refactor futur qui casserait la discrimination (par exemple en applatissant les deux branches du type) passerait silencieusement. Le seul filet restant est le `console.warn` runtime — utile mais ne bloque pas un build CI.

**Fix** : ajouter un vrai fichier `test_button_types.ts` (ou renommer en `.test-d.ts` pour tester avec [`tsd`](https://github.com/SamVerschueren/tsd) ou `vitest --typecheck`) contenant par exemple :

```ts
import Button from '../../../app/components/ui/Button.vue';

// Template compile-time : vue-tsc doit rejeter iconOnly sans ariaLabel.
// (Test via .vue fixture ou type assertion direct sur props.)
type Props = ComponentProps<typeof Button>;

// @ts-expect-error iconOnly=true requires ariaLabel:string
const _bad: Props = { iconOnly: true };

// @ts-expect-error iconOnly=true forbids ariaLabel=undefined
const _bad2: Props = { iconOnly: true, ariaLabel: undefined };

// OK : ariaLabel non-empty fourni
const _ok: Props = { iconOnly: true, ariaLabel: 'Fermer' };
```

Ou activer `vitest --typecheck` sur le fichier `.test-d.ts` dans le script `npm run test` (baseline tests inchangée au runtime, mais le typecheck échoue si `@ts-expect-error` ne trouve aucune erreur).

**HIGH-2 · WCAG 2.1 AA `color-contrast` non réellement vérifié + suspicion violation primary/danger**
Deux problèmes cumulés :

**(a) Harness inadéquat** — `test_button_a11y.test.ts` appelle `axe(document.body, AXE_OPTIONS)` avec un mount `@vue/test-utils` en environnement **happy-dom** sans pipeline CSS. Vitest ne charge pas Tailwind ni `main.css` dans le DOM de test. Les classes comme `bg-brand-green` sont bien présentes sur l'élément `<button>`, mais `getComputedStyle(element).backgroundColor` retourne `rgba(0, 0, 0, 0)` (aucune feuille de style résolue). Axe-core, ne pouvant calculer la couleur de fond réelle, retourne `incomplete` (ni violation ni pass) sur la règle `color-contrast`. Résultat : `toHaveNoViolations()` passe sur les 8 audits… sans jamais avoir effectivement audité le contraste.

**(b) Contrastes réels probablement en violation** — calcul manuel sur les tokens `@theme` `main.css:8-12` :
- Primary : `bg-brand-green (#10B981) text-white` → luminance relative 0,367 vs 1,0 → contraste **2,56:1** ❌ AA (requiert 4,5:1).
- Danger : `bg-brand-red (#EF4444) text-white` → **3,28:1** ❌ AA.
- Secondary : `bg-surface-bg (#F9FAFB) text-surface-text (#111827)` → ~17:1 ✅.
- Ghost : `bg-transparent text-surface-text` sur fond page surface-bg → ~17:1 ✅.

Les variants primary et danger **échouent** WCAG 2.1 AA `text-sm font-medium` (14 px, font-weight 500 ≠ « bold » au sens WCAG). La règle `color-contrast` axe-core détecterait cette violation en runtime (Storybook addon-a11y avec Tailwind compilé, navigateur réel), mais pas dans le harness Vitest actuel.

**Impact utilisateur** : CTA principaux (primary) et actions destructrices (danger) illisibles pour utilisateurs avec déficience visuelle légère — conflit direct avec la persona « PME AO mobile » (lumière ambiante forte, petits écrans) de la spec UX.

**Fix recommandé (ordre de préférence)** :
1. **Vérifier empiriquement** via Storybook runtime : `npm run storybook -> addon-a11y panel -> sélectionner Primary.Md` et consulter le rapport `color-contrast`. Si violation confirmée, exécuter l'option 2.
2. **Darken tokens** dans `main.css` (hors scope direct 10.15, mais candidat hotfix) : `--color-brand-green: #059669` (emerald-600, contraste ~3,91:1 — toujours marginal) ou `#047857` (emerald-700, ~5,77:1 ✅). Idem brand-red vers `#DC2626` (red-600, ~4,83:1 ✅).
3. **Ajouter `font-semibold`** aux variants primary/danger (font-weight 600) ? WCAG considère « large text » à partir de 14 px gras (≥ 700). `font-semibold` (600) reste zone grise — solution fragile.
4. **Court terme (accepter la dette)** : tracer dans `deferred-work.md` sous `DEF-10.15-2 Contraste AA primary/danger` avec échéance Phase 1 avant 1ʳᵉ démo externe. Parallèlement, upgrader le harness : exécuter un test E2E Storybook `storybook:test -- --ci` qui charge réellement les classes Tailwind compilées, ou ajouter un check `color-contrast` explicite en pré-commit sur les tokens eux-mêmes via un script dédié.

Cette décision conditionne l'usage de `ui/Button variant="primary"` pour les 60 migrations brownfield Epic 11-15 — la trancher avant la première migration FR40 SignatureModal évite un rollback coûteux.

**MEDIUM-4 · `disabled` HTML attribute bind sur `isInactive` coupe la focusabilité en loading**
`Button.vue:100` applique `:disabled="isInactive"` où `isInactive = disabled || loading`. En HTML natif, `<button disabled>` n'est **pas focusable au clavier** (comportement navigateur, pas un choix ARIA). Un bouton en état `loading=true` devient donc inaccessible à l'utilisateur qui voudrait explorer son état.

Les ARIA Authoring Practices (WAI-ARIA 1.2) recommandent pour un « busy button » :
- `aria-busy="true"` + bouton **focusable** (pas `disabled` HTML).
- Intercepter le clic via `@click.prevent` + early return (déjà fait dans `handleClick`).

Cela permet aux utilisateurs de clavier + lecteurs d'écran de parcourir le bouton loading, comprendre l'état via l'annonce « button, busy », et revenir dessus quand le spinner disparaît.

**Impact** : modéré, pas de violation WCAG stricte. Comportement actuel accepté par jest-axe et Storybook a11y addon (historiquement toléré).

**Fix** (à revisiter Story 10.21 avec Lucide Loader2) :
```vue
<button
  :disabled="disabled && !loading"    <!-- disabled HTML uniquement si vraiment disabled -->
  :aria-busy="loading ? 'true' : undefined"
  :aria-disabled="disabled || loading ? 'true' : undefined"
  ...
>
```

Tracer dans `deferred-work.md` comme `DEF-10.15-3 Loading button focusable AAA`.
