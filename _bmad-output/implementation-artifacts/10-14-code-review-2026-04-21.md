## Layer 1 — Code Quality Frontend

### HIGH

**HIGH-1 · Phantom dependency `@storybook/test`**
Les 6 `*.stories.ts` importent `@storybook/test` (`fn`, `expect`, `userEvent`, `within`) — cf. `SignatureModal.stories.ts:2`. Ce package **n'est pas déclaré** dans `frontend/package.json:32-53` ; il est résolu transitivement via `@storybook/addon-interactions` (confirmé `package-lock.json:5648,5824`).

**Risque** : un upgrade mineur d'addon-interactions qui supprimerait la re-export transitive casserait tous les imports, bloquant le build Storybook entier sans avertissement. Pratique contraire au principe de dépendances explicites.

**Fix** : ajouter explicitement dans `devDependencies` :
```json
"@storybook/test": "^8.6.18",
```

**HIGH-2 · `SourceCitationDrawer` utilise `role="dialog" aria-modal="true"` au lieu de `role="complementary"`**
`SourceCitationDrawer.vue:45-49` hérite de Reka UI `DialogRoot`/`DialogContent` qui impose `role="dialog"` + focus trap + Escape-to-close. Un drawer latéral de consultation de sources (FR71, traçabilité RAG) n'est **pas** un dialogue modal : l'utilisateur doit pouvoir parcourir la liste des sources tout en continuant à lire le contenu principal.

**Impact utilisateur (AT) :** un lecteur d'écran annonce « dialogue modal » et verrouille la lecture sur le drawer. Or la spec UX §3 parle d'un « panneau » consultable en parallèle du contenu principal. Le focus trap est contre-productif ici.

**Fix recommandé (ordre de préférence)** :
1. Remplacer par un élément natif `<aside role="complementary" aria-label="Source citation">` non portalisé avec bouton `Fermer` explicite (pas de focus trap natif nécessaire pour un complementary).
2. Ou conserver Dialog Reka UI mais forcer `modal={false}` (non modal) et revoir le `role` + absence de `aria-modal`.

Cette décision d'UX conditionne la logique d'intégration Epic 13 — trancher maintenant évite un refactor coûteux.

### MEDIUM

**MEDIUM-1 · Tests de rendu dégradés sur SignatureModal + SourceCitationDrawer**
`test_each_component_renders.test.ts:12-34` ne vérifie que `w.props('state')` pour ces deux composants, sans aucune assertion DOM. Le commentaire l.20 justifie par le Teleport Reka UI + happy-dom, mais la règle d'or 10.5 (rendu réel) citée dans `storybook.md:104-107` est contournée.

**Fix** : assertion minimale sur `document.body.querySelector('[role="dialog"]')` après `attachTo` + `nextTick`, ou expliciter la limite dans le test avec un `it.todo` ciblé pour ne pas masquer une régression future (ex. si Reka UI change le nom du rôle).

**MEDIUM-2 · Duplication de tokens couleurs `verdict` vs `brand`**
`main.css:8-12` et `main.css:22-30` définissent deux fois `#10B981` (`--color-brand-green` + `--color-verdict-pass`), `#EF4444` (`--color-brand-red` + `--color-verdict-fail`), `#F59E0B` (`--color-brand-orange` + `--color-verdict-reported`).

**Risque** : dérive visuelle si `brand` évolue (rebranding) sans que verdict suive. Contredit la leçon 10.5 « zéro duplication ».

**Fix** :
```css
--color-verdict-pass: var(--color-brand-green);
--color-verdict-fail: var(--color-brand-red);
--color-verdict-reported: var(--color-brand-orange);
```
Ou documenter explicitement dans un commentaire `main.css` que « verdict » est **sémantique** et **indépendant** de « brand » (choix délibéré, pas duplication accidentelle).

**MEDIUM-3 · A11y runtime non vérifiable hors exécution CI**
`test_a11y_axe.test.ts` couvre 4 composants sur 6 ; SignatureModal + SourceCitationDrawer sont délégués au runtime Storybook. La codemap §5 documente la dette (happy-dom ne matérialise pas Teleport de façon fiable). Acceptable **si** le job CI `Run a11y + interactions tests` (`storybook.yml:62-68`) échoue bien sur violation AA — mais aucun test ne prouve localement que le runtime exécute réellement axe sur ces deux composants.

**Fix** : ajouter un test CI explicite qui vérifie que l'audit addon-a11y a été exécuté sur `Gravity/SignatureModal` + `Gravity/SourceCitationDrawer` (par ex. grep sur log `test-storybook` pour la présence de ces noms de story).

**MEDIUM-4 · Absence de `tsc --noEmit` sur `.storybook/tsconfig.json` en CI**
`main.ts:14` désactive `typescript.check: false`. Combiné à `storybook.yml` qui n'exécute pas `tsc` dédié sur les stories, une erreur de type strict dans un `.stories.ts` ne bloque pas la CI (le build Storybook compile mais tolère). Incohérent avec l'exigence TS strict AC2.

**Fix** : ajouter un step CI :
```yaml
- name: Type-check Storybook stories
  run: npx tsc -p .storybook/tsconfig.json --noEmit
```

**MEDIUM-5 · Upgrade path `reka-ui` 3.0 non documenté**
`storybook.md:120-122` mentionne « revue trimestrielle » mais aucune référence à la roadmap Reka UI ni au plan d'audit breaking changes. Avec 6 composants gravity + usage prévu Epic 11-15 (ui/Dialog 10.15, ui/Drawer 10.16), un breaking Reka 3.0 a un blast radius élevé.

**Fix** : court ADR dans `docs/ADR/` ou note dans `deferred-work.md` listant les primitives Reka utilisées (`DialogRoot`, `ScrollArea`, `Tabs`) et la procédure upgrade.

### LOW

`.storybook/main.ts:15-24` injecte `@vitejs/plugin-vue` explicitement sans commenter pourquoi (le piège est documenté dans `storybook.md:111-116` mais pas à l'endroit où un contributeur futur le lira en premier).

**LOW-2 · `preview.ts:35-37` utilise des hex hardcodés pour les backgrounds**
`value: '#F9FAFB'` et `'#111827'`. Hors scope AC6 (scan gravity/*.vue uniquement), mais incohérent avec l'esprit « tokens only ».

**Fix** : référencer les tokens via `getComputedStyle` au runtime, ou documenter inline la raison (valeurs alignées sur `--color-surface-bg` / `--color-surface-dark-bg`).

**LOW-3 · `SectionReviewCheckpoint.vue:110-112` : `:disabled` + `:aria-disabled` redondants**
`disabled` HTML implique déjà l'état ARIA ; le doublon peut générer un warning axe sur certains utilitaires.

**Fix** : supprimer `:aria-disabled="!canExport"`.

**LOW-4 · `SgesBetaBanner.vue:69` : touch target `py-1` < 44px**
Le bouton « Demander la revue » fait ~26px de haut (py-1 + text-xs). Sous le seuil WCAG 2.1 AAA / standard mobile 44×44 (brief Layer 2 discipline Wave Q12).

**Fix** : `py-2 text-sm` ou `min-h-[44px]` explicite. AAA n'est pas bloquant MVP (AC4) mais la discipline reste active.

**LOW-5 · `ImpactProjectionPanel.vue:97-101` : `role="status"` sans `aria-live`**
Le bloc « Migration publiée » n'a pas `aria-live="polite"`. Or l'apparition de ce message suit l'action utilisateur « Publier » — un lecteur d'écran doit l'annoncer.

**Fix** : ajouter `aria-live="polite"` sur le div `role="status"` de publication réussie (cohérent avec `SourceCitationDrawer.vue:69` et `ReferentialComparisonView.vue:60`).

**LOW-6 · CI `storybook:test` invocation différente du script npm**
`storybook.yml:64-68` appelle `npx http-server` + `wait-on` + `npm run storybook:test -- --url http://…`, alors que le script `package.json:15` est juste `test-storybook`. Plus robuste en CI (http-server > file://) mais divergent du spec original AC5.

**Fix** : documenter ce choix dans la codemap §5 « Pièges documentés » (le pattern `file://` crash sur certaines ressources relatives).
